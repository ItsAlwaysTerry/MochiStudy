# MochiStudy 学习工作流审计报告

> 评估框架：**测水平 → 找差距 → 个性化讲解 → 出题 → 全景图**
>
> 核心原则：用 AI 做来回交互的学习，AI 是动态调整的老师，不是搜索引擎。
>
> 评估对象：App 各模块 + `skill/gaokao私教.md` + `skill/gaokao复习私教.md`

---

## 一、先理清楚这是一个什么系统

MochiStudy 不是独立存在的，它和两个 AI Prompt 共同构成一个三角闭环：

```
[gaokao私教.md]        ← 新学一道题（完整工作流：诊断→脚手架→套路→变式）
       ↓ 输出 MOCHI-RECORD
[MochiStudy App]       ← 记录结果 + 可视化进度 + 间隔复习触发
       ↓ 输出 AI复习包
[gaokao复习私教.md]    ← 读取历史卡点，出变式题，验证掌握
       ↓ 输出增强版 MOCHI-RECORD
[MochiStudy App]       ← 更新卡片 + 调整复习优先级 → 继续循环
```

**结论先说**：这个系统在设计层面是完整的，甚至是优秀的。  
**真正的问题是**：三角的三条边在用户体验上是断开的——每一个转换点都需要用户自己知道该怎么做，App 没有主动承担"连接者"的角色。

---

## 二、Prompt 评估：`gaokao私教.md`（AI 家教，新学模式）

### 工作流覆盖

| 工作流阶段 | 覆盖情况 | 对应 Prompt 步骤 |
|-----------|---------|----------------|
| 测水平 | ✅ | 第一步：3 状态诊断（空白型 / 方向错误型 / 卡步型） |
| 找差距 | ✅ | 第一步诊断结果 → 精准定位卡点 |
| 个性化讲解 | ✅ | 第二步 A/B/C：三种脚手架模式，按诊断结果分支 |
| 出题 | ✅ | 第五步：变式题（只改一个条件，同一套路） |
| 全景图 | ✅ | 第四步：3 步标准套路模型，学生先说 + AI 给标准版 |

**工作流完整度：5/5。这个 Prompt 完整实现了来回交互的学习工作流。**

### 亮点

- **3 状态诊断系统**是整个 Prompt 最精妙的设计。同一道题，「空白型」学生要从找已知条件开始；「卡步型」学生只需要突破那一步。这才是"动态调整的老师"的核心能力。

- **正反馈话术对照表**（说这个 vs 不说这个）是高质量的细节。把「你好棒」换成「你刚才想到用这个公式，这个方向完全正确」——这对一个长期被打压、缺乏自信的学生影响是巨大的。

- **变式题检验（第五步）** 正确实现了"间隔迁移"——不是重复原题，是用同一套路换一个条件，验证掌握的是方法而不是答案。

- **MOCHI-RECORD 输出强制要求**（"必须"、"一字不差"）确保数据回流，不会因为用户懒得复制而断链。

- **知识点约束到预设列表**是关键的数据工程决策——保证学习档案里的分类是可聚合的，不会出现"函数"和"函数问题"两条独立记录。

### 风险 / 问题

1. **用户入门门槛不低**：这个 Prompt 要作为 Claude Project 的 System Prompt 使用，或者每次对话开头粘贴。新用户不一定知道怎么操作。App 里没有任何指引告诉用户"去这里新建一个 AI 家教项目"。

2. **没有会话启动指南**：Prompt 很完整，但用户第一次打开 AI 对话时，应该发什么？发题目图片？直接说"我要学三角函数"？这个入口是缺失的。

3. **新学 vs 复习的分工不明确**：`gaokao私教.md` 的变式题（第五步）和 `gaokao复习私教.md` 功能有重叠。如果学生在家教 AI 里已经做了变式题，还需要再去复习 AI 那里重复吗？分工逻辑需要更清晰地传达给用户。

## 三、Prompt 评估：`gaokao复习私教.md`（AI 复习私教）

### 工作流覆盖

| 工作流阶段 | 覆盖情况 | 对应 Prompt 步骤 |
|-----------|---------|----------------|
| 测水平 | ✅ | 第二步：基于历史卡点生成测验题，测当前掌握程度 |
| 找差距 | ✅ | 第一步：优先级判断（低星、多次低星、久未碰、卡点具体） |
| 个性化讲解 | ✅ | 第三步：2 层提示系统（方向提示 → 关键步骤 → 才带着做） |
| 出题 | ✅ | 第二步：基于原题的同类变式，只针对一个卡点 |
| 全景图 | ✅ | 第四步：复盘套路（学生先说 + AI 给标准 3 步版） |

**工作流完整度：5/5。**

### 这是整个系统最精妙的部分

复习 AI 是**数据驱动的个性化**——它读的是 MochiStudy 学习档案，不是泛泛地出题，而是针对"你之前两次都卡在换元后忘记变量范围"出题。这才是真正的"动态调整"。

普通学生用 AI 复习：让 AI 出三角函数题目。
用这个系统复习：AI 看到你三角函数学了 4 次，2 次 1 星，知道你卡在和差化积，出一道专门训练这个点的题。

这是质量差距极大的两种复习体验。

### 亮点

- **优先级判断逻辑**（低星 > 多次低星 > 复习结果不是独立做对 > 久未碰）是合理的间隔重复策略，比单纯按时间排序更智能。

- **增强版 MOCHI-RECORD** 比家教 AI 的记录更丰富：多了 `复习结果 / 错误类型 / 卡住步骤 / 关键突破 / 题型标签 / 信心分 / 耗时分钟`。这些字段如果被 App 正确读取和聚合，能做出非常有价值的学情分析。

- **"不要把复盘变成讲座"**这条节奏原则很实际——防止 AI 惯性输出长篇解析，破坏学生主动思考的空间。

### 风险 / 问题

1. **增强版 MOCHI-RECORD 和 App 的字段对应关系存疑**：`study_log` 核心字段只有 7 个（不含 `学习来源`、`复习结果`、`错误类型`等），这些扩展字段是写入 `study_card_meta` 的。App 的导入解析器 `applyMochiRecord` 是否实际解析并保存了这些扩展字段？如果没有，这些字段就是数据黑洞——用户填了，但 App 没存，复习包里也就没有这些信息了。**这是最需要验证的技术断点。**

2. **"AI 复习包"的实际格式不透明**：Prompt 说"用户会粘贴 AI 复习包"，但这个包长什么样，App 里的 `copyItemPack()` 实际生成什么格式，Prompt 里没有示例。AI 在不知道格式的情况下，可能读不准或忽略某些字段。

3. **单次只复习 1-2 个知识点**的限制合理，但如果用户有 10 个到期知识点，该怎么选？Prompt 有优先级逻辑，但没有说"如果时间多，可以做第二轮"。

## 四、App 模块逐一评估

### 4.1 导入记录（`parsePastedRecordEl` / MOCHI-RECORD 解析）

**工作流定位**：捕获找差距和出题的结果。

这是 App 最正确的设计决策。让 AI 输出结构化格式，App 解析保存，解耦了 AI 交互和数据持久化。

**问题**：
- App 只管收，没有告诉用户怎么产生这份记录
- 导入框有提示文字，但没有"去哪里获取这份记录"的链接或说明

**打分：7/10**

---

### 4.2 复习页（`reviewPage.js` + `reviewEngine.js`）

**工作流定位**：间隔复习触发，应该是"测水平 + 出题"的入口。

按 `docs/review-module.md` 的设计，这是整个系统的调度中心。复习页读取 `study_log + study_card_meta`，聚合知识点状态，生成"AI 复习包"，用户拿去粘贴给复习 AI，回来粘贴结果。

**亮点**：
- "先自己回想 20 秒"的提示是正确的主动回忆设计
- 间隔复习算法（低星、久未碰、上次未做对）和复习 AI 的优先级逻辑是一致的

**问题**：
- 用户点"复制材料"后，没有任何提示说"粘贴给复习 AI（链接或说明在这里）"
- 如果用户不知道 `gaokao复习私教.md` 的存在，这个复制按钮的意义就为零
- 复习包的实际内容格式——用户看不到自己复制了什么，是否包含了 AI Prompt 所需的所有字段？

**打分：4/10（功能在，但连接断了）**

---

### 4.3 学习档案（`knowledgeMap.js`）

**工作流定位**：全景图——让学生感知整体知识覆盖。

三层结构（科目→知识点→卡片），状态标记（untouched / learning / mastered / dormant），是全景图的雏形。

**问题**：
- 没有因果分析——为什么"函数"一直 learning？哪些卡点反复出现？
- 卡片是时间线堆叠，用户需要自己阅读所有卡片才能得到规律
- `docs/review-module.md` 里的"核心摘要卡"设计（把历史卡沉淀成精华摘要）还没完全实现，用户现在面对的是原始卡片堆

**`study_node_summary` 的人工摘要**是当前最接近全景图洞察的功能，但 UI 上是否足够醒目？

**打分：5/10（有形状，缺洞察）**

---

### 4.4 番茄钟（`timer.js`）

**工作流定位**：时间结构，和学习内容完全解耦。

Deciding 阶段（结束→选开始休息/继续）符合认知科学，避免立刻切换。

**问题**：计时器开始前，App 完全不知道用户在学什么，也不会建议用户去哪里开始。两个系统（番茄钟和学习内容）是平行的，没有整合。

**打分：6/10（计时器本身好，但孤立于学习流程）**

---

### 4.5 农场系统（`farm.js` 农场部分）

**工作流定位**：无，纯动机系统。

对目标用户（基础差、缺自信、喜欢收集）是正确的选择。把抽象的"学了多少"变成可见的农作物生长，提供情绪价值。

**当前问题**：recordCount 驱动生长，1 星和 3 星对农场贡献相同。这是合理的取舍（不想让学生因为低星觉得"连农场都没用"），但意味着农场无法反映学习质量。

**打分：N/A（不该用学习工作流框架评估，动机设计本身是对的）**

---

### 4.6 Streak 横幅 + 今日复习卡（`renderStreakBanner` / `renderTodayReviewCard`）

**工作流定位**：首页入口，低门槛触发复习。

"今日复习"卡（`<details>` 遮住卡点，主动揭开）是主动回忆设计，很好。

**问题**：这张卡现在只是一个入口，点击"复制材料"后用户还是会迷失在"然后呢"的问题里。

---

### 4.7 勋章 + 抽奖（`app.js`）

**工作流定位**：无，奖励系统。

勋章逻辑简单清晰（小勋章/大勋章按阈值触发）。抽奖实现复杂（牌局流程、Canvas 分享、木鱼……）。

**需要问的问题**：抽奖系统的开发和维护成本是否值得？它解决了什么用户问题（短期娱乐 / 奖励循环），而这个问题用更简单的实现是否也能解决？

---

### 4.8 赛季系统（`app.js`）

**工作流定位**：长期进度可视化。

热力图和趋势图只统计数量，不统计质量（平均星级、卡点解决率）。赛季数据告诉用户"你学了多努力"，但不告诉用户"你学得多好"。

---

## 五、整体闭环分析

### 系统在设计上是完整的

```
[测水平]   → gaokao私教.md 的第一步诊断
[找差距]   → 诊断结果 + 卡点记录 → MOCHI-RECORD 保存
[个性化讲解] → 脚手架模式（3种）+ 复习私教的 2 层提示
[出题]     → 变式题 + 复习私教的复习题
[全景图]   → 学习档案（知识点状态）+ 套路模型（第四步输出）
```

五个阶段都有对应的实现。这是一个设计上优秀的学习系统。

### 但用户体验是断裂的

用户需要自己知道以下事情，App 没有主动告知：

```
断点 1：学新题时 → 去哪里找 gaokao私教.md？怎么设置 Claude Project？
断点 2：做完题后 → MOCHI-RECORD 怎么复制进 App？
断点 3：复习时 → 复习包复制后粘贴到哪个 AI？用哪个 Prompt？
断点 4：复习 AI 给出 MOCHI-RECORD 后 → 怎么导回 App？
```

每一个断点都依赖用户的先验知识。用户每次都在脑子里手动完成系统设计图里的那条箭头。

## 六、改进建议（按优先级）

### 🔴 P0：在 App 里把 AI 工作流的入口显性化

**具体做什么**：在首页或复习页加一个"使用说明"入口，展示：

- 第一步：[点击这里] 打开 AI 家教（链接到 Claude，或直接展示 Prompt）
- 第二步：把题目图片发给 AI，按 AI 的引导做题
- 第三步：AI 会自动生成学习记录，复制后粘贴进这里 [指向导入框]
- 复习时：[点击这里] 打开复习 AI（链接 + Prompt），把复习包粘贴过去

这不需要改任何技术，就是一段说明文字 + 链接。但对新用户的影响是巨大的。

---

### 🔴 P0：验证增强版 MOCHI-RECORD 字段是否真的被 App 解析存储

`gaokao复习私教.md` 输出的记录包含 `复习结果 / 错误类型 / 卡住步骤 / 关键突破 / 题型标签 / 信心分 / 耗时分钟`。

需要确认 `parsePastedRecordEl()` / `applyMochiRecord()` 是否实际解析这些字段并存入 `study_card_meta`。

如果没有，这些字段是数据黑洞：用户填了，AI 写了，但 App 没保存，下次复习包里也就没有这些信息，复习 AI 的数据驱动优势就消失了。

**验证方法**：导入一条包含全部扩展字段的 MOCHI-RECORD，然后检查 localStorage 里的 `study_card_meta`。

---

### 🔴 P0：为复习包加上 Prompt 指引文本

当前"复制材料"复制的是卡点/原题/套路的纯数据。  
改成：复制的内容 = 指引文本 + 数据包：

```
你是复习 AI 私教。以下是我的学习档案，请按你的工作流程为我安排今天的复习。

===== 学习档案开始 =====
科目：数学
知识点：函数
上次学习：2026-05-20（星级：1★）
卡点：换元后忘记检查新变量的范围
历史套路：第一步 换元...
===== 学习档案结束 =====
```

这样用户粘贴进 AI 就能直接开始，不需要额外设置 System Prompt。

---

### 🟡 P1：学习档案加"卡点聚合"视图

当前：每个知识点展开后是时间线卡片堆叠。

改进：展开知识点时，先显示**规则生成的摘要**（`study_node_summary` 已有此设计）：

```
函数 · 学习中（4次，1-2星）

📌 主要卡点（出现 3 次）：换元后忘记检查变量范围
🔑 最近突破：无
📋 下一步建议：已到复习时间，建议今天复习

▼ 展开历史卡片（4张）
```

这是真正的全景图：告诉用户不只是"我学过什么"，而是"我在哪里卡住，卡了多少次"。

---

### 🟡 P1：番茄钟开始前显示"今天建议学什么"

开始计时前，显示：

```
建议今天：复习「函数」（距上次 5 天，1 星记录）
[复制复习包，去 AI 那里开始] →
```

这把番茄钟和学习内容连接起来，番茄钟从"计时器"变成"学习会话启动器"。

---

### 🟡 P1：App 内展示 Prompt 副本

把 `gaokao私教.md` 和 `gaokao复习私教.md` 的内容放进 App 的"设置 → 使用指南"里，让用户可以在 App 内复制 Prompt，不需要单独找文件。

---

### 🟢 P2：赛季数据加星级质量维度

热力图加颜色深浅表示平均星级，而不只是"有没有记录"。1 星记录是浅绿，3 星记录是深绿，让学生能感知"努力了但质量低"和"学得扎实"的区别。

---

### 🟢 P2：复习包加"上次复习 AI 的关键突破"字段

如果增强版 MOCHI-RECORD 的 `关键突破` 字段被正确存储，下次复习包应该优先把这条突破展示给复习 AI：

```
上次复习结果：看提示做对（2026-05-18）
上次关键突破：知道了要在换元后立刻写新变量范围
本次建议：验证是否能独立做到
```

这才是真正的"来回交互"——每次复习都建立在上一次突破的基础上。

## 七、建议降优先级 / 简化的部分

### 抽奖系统

抽奖牌局是整个 App 代码量最重的功能，但对学习工作流贡献为零。Canvas 转盘动画、翻牌效果、分享图片生成……这些细节的开发成本极高。

**不建议删除**（它有娱乐价值，用户喜欢），但**建议暂停迭代**。把相同的时间投入到复习包 + AI 引导上，对学习体验的提升会远大于把抽奖做得更精美。

### 赛季系统热力图

目前热力图只是好看，数据太浅（只有"有没有记录"）。如果不加质量维度，这是装饰性功能。可以先冻结，等赛季数据有质量字段后再升级。

### 日历页

低频功能，已经在次级导航。长期不动没有问题。

## 八、总结：这个系统的真实水位

| 维度 | 水位 |
|------|------|
| 学习工作流设计 | ⭐⭐⭐⭐⭐ 两个 Prompt 完整实现了5步工作流 |
| 数据闭环设计 | ⭐⭐⭐⭐ 架构清晰，MOCHI-RECORD 桥接优雅，但字段存储需验证 |
| 用户体验连贯性 | ⭐⭐ 三条边之间的转换全靠用户自己跨越 |
| 动机/游戏化 | ⭐⭐⭐⭐ 对目标用户画像非常合适 |
| 可维护性 | ⭐⭐⭐ 抽奖系统过重，其他模块轻量 |
| 学情分析深度 | ⭐⭐ 现在只统计数量，质量数据有但未充分利用 |

> **一句话定位**：这是一个设计优秀但用户体验断裂的学习系统。核心价值（AI 工作流 + 间隔复习 + 数据闭环）完全正确，最高优先级的工作是把三条边的连接点显性化——让用户在 App 里就能看到"下一步去哪里，带什么，回来粘贴在哪里"。

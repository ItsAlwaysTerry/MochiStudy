# MochiStudy 题桌 PRD

日期：2026-06-18

## 1. 背景

MochiStudy 目前更像一个“学完之后的记录系统”：学生在外部 AI 或其他学习场景里完成一道题，再把 AI 输出的 MOCHI-RECORD 粘回网站，网站负责学习档案、复习队列、农场、勋章和今日报告。

这个流程的问题是：真正发生学习的地方不在 MochiStudy 里。学生需要在题目图片、外部 AI、MochiStudy 导入框之间来回切换，操作链路长，记录也依赖学生主动复制粘贴。

“题桌”的目标是把学习现场搬进 MochiStudy：学生把题目图片粘贴进来，在图片上框选题目，直接在题目旁边问 AI，问完后在原图位置留下标记，并一键沉淀到现有学习档案。

## 2. 产品定位

题桌是 MochiStudy 的主学习工作台。

它不是无限白板，也不是传统文件管理器，而是：

- 左侧：题目文件栏，用来管理粘贴进来的题图。
- 中间：当前题图/试卷页，学生在这里框题、看题、回到原题位置。
- 右侧：题目级学习面板，承载 AI 对话、学习总结、保存到学习档案。

一句话定义：

> 学生把题图粘进题桌，在原题位置开一个 AI 学习空间，学完自动变成 MochiStudy 的学习卡片。

## 3. 目标用户与场景

目标用户是基础薄弱、需要高频陪伴式讲题的高中生。

真实使用场景：

- 学生做练习册，遇到不会的题，直接截图或拍照粘进题桌。
- 学生拍半页卷子，框选其中一道题问 AI。
- 学生后期可能向老师要 PDF 版试卷，再导入题桌逐题学习。
- 学生不想先整理资料，但希望问过的题以后能找回来。
- 家长或老师希望看到学生今天到底搞懂了哪些题、哪些知识点。

## 4. 产品原则

### 4.1 先学习，后整理

题图先进入收件箱，不能要求学生上传时就完整填写科目、知识点、标签。整理应该发生在保存学习记录时，或者由 AI 自动建议。

### 4.2 题目是最小单位

系统核心对象不是“笔记”或“试卷”，而是一道被框选、被问过、被记录的题目。

### 4.3 题旁留痕

只要一道题发生过 AI 学习或保存过记录，就必须在原图对应位置留下轻量标记。点击标记可以回看这道题的对话、总结和学习卡片。

### 4.4 记录自动化

学生不应该再手动复制 MOCHI-RECORD。题桌内部可以暂时保留“粘贴 AI 输出”作为过渡，但产品方向是：AI 对话结束后生成学习记录草稿，学生确认即可保存。

### 4.5 主学习区成熟清爽

题桌视觉不沿用首页的游戏感。它应该更像清爽的笔记软件或问答工作台：浅灰背景、白色纸面、轻工具栏、少量蓝色强调、清晰留白、低装饰。

## 5. 非目标

第一阶段不做以下内容：

- PDF 导入和多页 PDF 渲染。
- OCR 自动识别题目边界。
- 手写批注、画笔、橡皮、完整 GoodNotes 级标注系统。
- 复杂自定义题本、全文搜索、批量整理。
- 真实多模型 AI 工作流的完整优化。
- 修改现有 `study_log` 字段结构。
- 推翻现有农场、学习档案、复习队列、勋章系统。

这些是后续阶段的方向，不属于第一轮验证。

## 6. 信息架构

### 6.1 主导航

新增主导航项：

- 题桌

建议位置：放在“首页”和“学习”之间，作为新的主学习入口。

### 6.2 题桌页面结构

```text
------------------------------------------------------+

| 数学         | 框选区域              | AI 对话/记录     |

| 已学习       |                      |                 |

```

移动端：

- 默认显示中间题图。
- 左侧文件栏变为抽屉。
- 右侧学习面板变为底部弹层或全屏面板。

## 7. 核心用户路径

### 7.1 粘贴题图

1. 学生复制一张图片。
2. 回到题桌，按 `Ctrl+V` 或点击“粘贴题图”。
3. 系统读取剪贴板图片。
4. 弹出轻量命名条，默认名称已生成并全选。
5. 学生直接回车保存，或输入新名称覆盖。
6. 图片进入左侧文件栏，并在中间打开。

### 7.2 自动命名

默认命名格式：

```text
科目-类型-YYYY-MM-DD-编号
```

例子：

```text
数学-周测-2026-06-18-01
数学-周测-2026-06-18-02
物理-练习册-2026-06-18-01
未分类-拍题-2026-06-18-01
```

命名规则：

- 默认科目为“未分类”。
- 默认类型为“拍题”。
- 如果用户上一次使用了同一天、同科目、同类型的名称，下一次自动递增编号。
- 命名输入框默认全选，用户直接打字即可替换。
- 用户手动改名后，系统尝试从新名称中提取科目、类型、日期、编号，作为下一次命名模板。
- 左侧文件栏显示短名称，例如 `数学-周测-06-18-02`。

### 7.3 框选题目

1. 学生打开一张题图。
2. 点击“框题”或直接在图片上拖拽。
3. 图片上出现蓝色虚线/实线选区。
4. 选区旁出现轻浮条：

```text
问 AI | 整张作为一题 | 保存区域 | 取消
```

5. 如果图片本来就是一道题，学生可以点击“整张作为一题”。

### 7.4 打开 AI 学习面板

1. 学生点击“问 AI”。
2. 右侧学习面板打开。
3. 面板显示：
   - 题目截图。
   - 学生输入框。
   - AI 对话区。
   - 学习记录草稿区域。
4. 第一阶段可以先不接真实 AI，允许粘贴 AI 输出或手动填记录草稿。
5. 第二阶段接入真实 AI 后，使用 `skill/gaokao啃卷子.md` 的题目讲解规则作为系统提示基础。

### 7.5 题旁标记

当题目区域创建后，原图上出现一个小圆标：

- 未学习：浅灰或空心标记。
- 已问 AI：蓝色标记。
- 已保存学习档案：蓝色实心标记，可带小勾。

点击小圆标：

- 打开右侧学习面板。
- 回看题目截图、对话、学习记录草稿、保存状态。

### 7.6 保存到学习档案

题目面板生成或填写学习记录草稿后，学生点击“保存到学习档案”。

保存后：

- 调用现有 `window.MochiApp.applyMochiRecord(record)`。
- 写入现有 `study_log`。
- 写入现有 `study_card_meta`。
- 刷新学习档案、复习队列、农场、勋章、今日学习报告。
- 当前题目状态变为“已学习”。
- 原图小圆标更新为“已保存”状态。

## 8. 数据模型草案

### 8.1 存储原则

图片本体不存 localStorage。图片文件放 IndexedDB，localStorage 只存轻量索引和题目元数据。

原因：

- localStorage 容量小，通常只有 5MB 左右。
- 图片转 base64 后体积大，容易爆容量。
- IndexedDB 更适合存 Blob。

### 8.2 IndexedDB

数据库名建议：

```text
mochi_question_desk
```

对象仓库：

```text
question_images_blob
```

保存：

```javascript
{
  id: "img_...",
  blob: Blob,
  mimeType: "image/png",
  createdAt: ISOString
}
```

### 8.3 localStorage keys

新增：

```text
question_desk_images
question_desk_questions
question_desk_name_template
question_desk_ui_state
```

### 8.4 question_desk_images

```javascript
[
  {
    id: "img_1718700000000_abcd",
    name: "数学-周测-2026-06-18-01",
    shortName: "数学-周测-06-18-01",
    subject: "math" | "physics" | "chemistry" | "uncategorized",
    sourceType: "拍题" | "周测" | "练习册" | "试卷" | "其他",
    date: "YYYY-MM-DD",
    mimeType: "image/png",
    width: 1280,
    height: 960,
    createdAt: ISOString,
    updatedAt: ISOString,
    questionCount: 0,
    learnedCount: 0
  }
]
```

### 8.5 question_desk_questions

```javascript
[
  {
    id: "q_1718700000000_abcd",
    imageId: "img_1718700000000_abcd",
    rect: {
      x: 0.12,
      y: 0.24,
      width: 0.72,
      height: 0.20
    },
    marker: {
      x: 0.10,
      y: 0.24
    },
    subject: "math" | "physics" | "chemistry" | "uncategorized",
    nodeLabel: "",
    status: "selected" | "asked" | "saved",
    title: "第1题",
    questionImageBlobId: "crop_...",
    chat: [
      {
        role: "user" | "assistant",
        content: "...",
        createdAt: ISOString
      }
    ],
    recordDraft: {
      subject: "math",
      nodeLabel: "函数",
      stars: 2,
      painPoint: "...",
      originalQuestion: "...",
      routine: "...",
      date: "YYYY-MM-DD",
      meta: {
        source: "lesson",
        errorType: "审题",
        keyInsight: "...",
        confidence: 3,
        timeSpentMinutes: 12
      }
    },
    savedLogId: "",
    createdAt: ISOString,
    updatedAt: ISOString
  }
]
```

坐标使用 0-1 的相对坐标，避免图片缩放后标记错位。

### 8.6 备份兼容

第一阶段暂不修改现有备份结构的 `version` 和 `data` 主结构。

后续如果要纳入完整备份，应把新增 localStorage key 纳入 `data.localStorage` 快照即可。IndexedDB 图片是否进入备份需要单独设计，因为图片体积大。

## 9. 分阶段路线

### Phase 0：PRD 与技术验证

目标：确认题桌关键技术可行。

验证点：

- 能监听粘贴事件并读取剪贴板图片。
- 能将图片 Blob 存入 IndexedDB。
- 能从 IndexedDB 读取 Blob 并显示。
- 能在图片上拖拽框选区域。
- 能根据相对坐标裁出题目截图。
- 能将题目元数据存入 localStorage。
- 能调用现有 `applyMochiRecord()` 生成学习卡片。

产出：

- 技术 spike 或小 demo。
- 不要求完整 UI。

### Phase 1：题桌 MVP

目标：跑通“粘贴题图 → 框题 → 留标记 → 保存学习档案”的核心闭环。

范围：

- 新增导航「题桌」。
- 题桌清爽三栏布局。
- 支持粘贴图片。
- 支持拖拽/点击上传图片作为兜底。
- 粘贴后自动命名，名称可编辑，编号自动递增。
- 左侧基础文件栏：
  - 收件箱
  - 数学
  - 物理
  - 化学
  - 已学习
  - 未学习
- 中间单图查看器。
- 支持框选题目区域。
- 支持“整张作为一题”。
- 支持题旁小圆标。
- 右侧题目面板。
- 第一版 AI 面板可先用“粘贴 AI 输出 / 手动填写记录草稿”的方式验证流程。
- 保存到学习档案并刷新现有系统。

成功标准：

- 学生可以在 30 秒内把一张题图粘进题桌并开始框题。
- 一道题可以从原图位置打开学习面板。
- 学完后可以保存为现有学习档案中的一张卡片。
- 复习队列和今日学习报告能看到这条记录。

### Phase 2：站内 AI 对话

目标：替代外部智能体。

范围：

- 使用现有 AI API 配置。
- 题目截图作为上下文发送给 AI。
- 使用 `gaokao啃卷子` prompt 作为题目讲解规则基础。
- 支持多轮对话。
- AI 自动生成学习记录草稿。
- 学生确认后保存。

成功标准：

- 学生不需要离开 MochiStudy 就能完成一道题的问答。
- 不再需要复制 MOCHI-RECORD 回来。

### Phase 3：题本与整理能力

目标：解决长期使用后的资料管理问题。

范围：

- 自定义题本。
- 重命名题图。
- 移动题图到题本。
- 按科目、日期、是否已学习、知识点过滤。
- 搜索题图名称和题目标题。
- 批量删除或归档。

成功标准：

- 使用一个月后，学生仍能找回问过的题。

### Phase 4：PDF 支持

目标：支持正式试卷和电子资料。

范围：

- 上传 PDF。
- 多页预览。
- 每页像图片一样框选题目。
- 题目锚点绑定到 PDF 页码和相对坐标。
- 左侧文件栏支持 PDF 展开页列表。

成功标准：

- 一份 PDF 试卷可以逐页框题并保存学习记录。

### Phase 5：学习橱窗与反向跳转

目标：让题旁标记成为长期学习资产入口。

范围：

- 点小圆标展示题目截图、AI 对话、卡点、套路、保存状态、复习历史。
- 从学习档案卡片反向跳回原题位置。
- 今日学习报告展示题目截图。
- 复习时可以直接打开原题图。

成功标准：

- 学习档案不只是文字卡片，而能回到原始题目现场。

## 11. Phase 1 验收标准

功能验收：

- 可以通过粘贴图片创建题图。
- 图片进入左侧文件栏，名称不是乱码。
- 连续粘贴同模板图片时，编号自动递增。
- 可以点击左侧文件打开图片。
- 可以框选一道题。
- 可以将整张图作为一道题。
- 框选后出现题旁小圆标。
- 点击小圆标打开题目学习面板。
- 可以填写或粘贴学习记录草稿。
- 点击保存后，现有学习档案出现对应卡片。
- 农场、复习队列、勋章状态能正常刷新。

体验验收：

- 桌面端三栏布局清晰。
- 主视觉清爽，不像小游戏页面。
- 学生不需要理解文件系统也能完成一次学习。
- 不上传到服务器，仍保持本地前端应用特性。

技术验收：

- 图片不写入 localStorage。
- JS 文件通过 `node --check`。
- 不修改 `study_log` 单条记录字段结构。
- 不修改备份文件 `version` 和主结构。
- 不引入后端依赖。

## 12. 风险与待确认问题

### 12.1 剪贴板兼容性

不同浏览器对读取剪贴板图片支持不完全一致。需要保留拖拽和点击上传作为兜底。

### 12.2 IndexedDB 备份

当前备份机制主要覆盖 localStorage。IndexedDB 图片如果不进入备份，用户换设备会丢失原题图。第一阶段可以接受，但 PRD 中需要标记为后续问题。

### 12.3 AI 图片理解能力

Phase 2 需要确认现有 API 配置是否支持图片输入。如果不支持，需要先把题目截图转成可见附件或让用户补充文字描述。

### 12.4 移动端框选体验

手机或平板上的拖拽框选需要单独优化，避免和页面滚动冲突。

### 12.5 学科自动识别

第一阶段不依赖自动识别。保存记录时用学生选择或 AI 输出的科目来归类。

## 13. 当前决策

- 新页面命名：题桌。
- 第一版输入方式：以粘贴图片为主，拖拽和上传为兜底。
- 第一版资料形态：左侧文件栏 + 中间单图查看器 + 右侧学习面板。
- 第一版不做无限画布。
- 第一版不做 PDF。
- 第一版必须有基础文件栏，不能等到后续阶段才整理。
- 第一版目标是验证学习闭环，不追求完整 GoodNotes 级体验。

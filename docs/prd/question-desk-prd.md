# MochiStudy 题桌 PRD

日期：2026-06-18

## 1. 背景

MochiStudy 目前更像一个“学完之后的记录系统”：学生在外部 AI 或其他学习场景里完成一道题，再把 AI 输出的 MOCHI-RECORD 粘回网站，网站负责学习档案、复习队列、农场、勋章和今日报告。

真实摩擦不是“缺一个上传题图的地方”，而是学习现场不在 MochiStudy 里。学生需要在题图、外部 AI、MochiStudy 导入框之间来回切换，操作链路长，记录也依赖学生主动复制粘贴。

题桌要解决的是：学生看到题不会做的那一刻，可以直接把题图粘进 MochiStudy，在站内问 AI，问完后把学习结果保存进现有学习档案。

第一版必须上线当天就比现状更省事。不能先做一套很重的图片管理和框选系统，却仍然要求学生去外部 AI 复制粘贴。站内视觉 AI 对话是题桌成立的前提，必须前置验证并进入第一版闭环。

## 2. 产品定位

题桌是 MochiStudy 的主学习工作台。

第一阶段它不是无限白板，也不是 GoodNotes 级批注工具，而是：

- 左侧：题图文件栏，用来管理粘贴进来的题目图片。
- 中间：当前题图查看器，一张图默认就是一道题。
- 右侧：站内 AI 学习面板，承载题图、对话、学习记录草稿、保存到学习档案。

一句话定义：

> 学生把一道题图粘进题桌，在站内让 AI 带着学，学完自动变成 MochiStudy 的学习卡片。

后续阶段再扩展为“一页卷子框多题”“题旁标记”“PDF 试卷”“从学习档案反跳回原题现场”。

## 3. 目标用户与场景

目标用户是基础薄弱、需要高频陪伴式讲题的高中生。

真实使用场景：

- 学生做练习册，遇到不会的题，直接截图或拍照粘进题桌。
- 学生日常更多是零散上传一道题，而不是先整理完整试卷。
- 学生偶尔会拍半页卷子或一页卷子，但第一版仍建议让他先按“一图一题”的方式学习。
- 学生后期可能向老师要 PDF 版试卷，再导入题桌逐题学习。
- 学生不想先整理资料，但希望问过的题以后能找回来。
- 家长或老师希望看到学生今天到底搞懂了哪些题、哪些知识点。

## 4. 产品原则

### 4.1 第一版必须比旧流程短

旧流程是：

```text
截图 -> 外部 AI -> 复制 MOCHI-RECORD -> 粘进导入框
```

题桌 Phase 1 必须变成：

```text
粘图 -> 站内 AI 问答 -> 确认保存
```

如果第一版仍要求“粘图 -> 去外部 AI -> 复制输出 -> 粘回题桌”，它就不是 MVP，而是比现状更累的中间态。

### 4.2 一图一题先跑通

第一版不做框选、裁剪和题旁标记。一张图片默认就是一道题。这样更符合学生日常“遇到一道题就拍一道题”的真实输入方式，也能把工程精力集中在站内 AI 和保存闭环上。

### 4.3 先学习，后整理

题图先进入收件箱，不能要求学生上传时就完整填写科目、知识点、标签。整理应该发生在保存学习记录时，由 AI 输出和学生确认共同完成。

### 4.4 题目是最小单位

系统核心对象不是“笔记”或“试卷”，而是一道被问过、被讲过、被保存的题目。

### 4.5 清爽界面，但保留成长反馈

题桌的学习区要清爽、成熟、低干扰：浅灰背景、白色纸面、轻工具栏、少量蓝色强调。

但保存成功那一刻必须保留 MochiStudy 的动力引擎：农场成长、sparkle、连续学习、勋章或今日进度反馈。题桌不能把“学完看得见成长”的闭环藏起来。

### 4.6 记录自动化

学生不应该再手动复制 MOCHI-RECORD。AI 对话结束后，系统生成学习记录草稿，学生确认即可保存到现有 `study_log` 和 `study_card_meta`。

## 5. 非目标

Phase 1 不做以下内容：

- PDF 导入和多页 PDF 渲染。
- OCR 自动识别题目边界。
- 半页/整页卷子的框选、裁剪、坐标标记。
- 题旁小圆标和从原图位置回看对话。
- 手写批注、画笔、橡皮、完整 GoodNotes 级标注系统。
- 复杂自定义题本、全文搜索、批量整理。
- 从用户手动改名中反推命名模板。
- 跨题“啃卷子”优先级排序。
- 修改现有 `study_log` 字段结构。
- 推翻现有农场、学习档案、复习队列、勋章系统。

这些是后续阶段的方向，不属于第一轮验证。

## 6. 信息架构

### 6.1 主导航策略

题桌不直接新增第 5 个底部主导航。

原因：MochiStudy 之前已经刻意把移动端底部导航收敛为“首页 / 学习 / 勋章 / 设置”，题桌如果直接再加一个主 tab，会反向增加导航负担。

Phase 1 建议：

- 桌面侧边栏可以在“学习”下突出题桌入口，或将学习页默认子 tab 改为“题桌”。
- 移动端仍保留 4 个底部导航，题桌作为“学习”里的第一子页。
- 如果题桌验证成功，后续再讨论它是否替代现有“学习”tab，而不是并排新增长期主导航。

### 6.2 Phase 1 页面结构

桌面端：

```text
+------------------------------------------------------+
| 顶部轻工具栏：粘贴题图 / 上传兜底 / 导出题桌包        |
+-------------+----------------------+-----------------+
| 左侧文件栏   | 中间题图查看器         | 右侧 AI 学习面板 |
| 收件箱       | 当前题图              | 题目截图         |
| 数学         | 一图一题              | 多轮对话         |
| 物理         |                      | 记录草稿         |
| 化学         |                      | 保存到学习档案   |
| 已学习       |                      | 成长反馈         |
| 未学习       |                      |                 |
+-------------+----------------------+-----------------+
```

移动端：

- 默认显示题图和 AI 学习入口。
- 左侧文件栏变为抽屉。
- 右侧学习面板变为底部弹层或全屏面板。

## 7. 核心用户路径

### 7.1 粘贴题图

1. 学生复制一张图片。
2. 回到题桌，按 `Ctrl+V` 或点击“粘贴题图”。
3. 系统读取剪贴板图片。
4. 弹出轻量命名条，默认名称已生成并全选。
5. 学生直接回车保存，或输入新名称覆盖。
6. 图片进入左侧文件栏，并在中间打开。
7. 右侧 AI 学习面板自动准备好这张题图。

### 7.2 Phase 1 自动命名

默认命名格式：

```text
未分类-拍题-YYYY-MM-DD-编号
```

例子：

```text
未分类-拍题-2026-06-18-01
未分类-拍题-2026-06-18-02
```

Phase 1 命名规则：

- 默认科目为“未分类”。
- 默认类型为“拍题”。
- 同一天连续粘贴时，编号自动递增。
- 命名输入框默认全选，用户直接打字即可替换。
- 用户可以手动选择科目，但系统不从自定义名称里反推模板。
- 左侧文件栏显示短名称，例如 `未分类-拍题-06-18-02`。

后续整理阶段再支持“数学-周测-日期-编号”等模板记忆和反推。

### 7.3 站内 AI 学习

1. 学生打开一张题图。
2. 右侧面板显示题图缩略图和输入框。
3. 学生输入“这题怎么做”“我看不懂第一步”等问题。
4. 系统把题图和学生问题发给支持图片输入的 AI。
5. AI 按“逐题讲解”规则引导学生，不直接只给答案。
6. 支持多轮追问。
7. 对话结束后，学生点击“生成学习记录”或 AI 自动生成记录草稿。

Phase 1 不能依赖外部 AI 复制粘贴。如果 Phase 0 证明当前 API 配置不能处理图片输入，题桌进入 no-go 或先做 API 配置升级，不发布假闭环。

### 7.4 保存到学习档案

题目面板生成学习记录草稿后，学生点击“保存到学习档案”。

保存前必须确认：

- 科目必须是数学 / 物理 / 化学之一。
- 知识点必须来自现有预设列表。
- 星级为 1-3。
- 原题描述、卡点、今日套路有合理内容。

保存时：

- 将 `nodeLabel` 解析成现有 `nodeId`。
- 调用现有 `window.MochiApp.applyMochiRecord(record)`。
- 写入现有 `study_log`。
- 写入现有 `study_card_meta`。
- 保存 `savedLogId` 回题桌题目记录。
- 刷新学习档案、复习队列、农场、勋章、今日学习报告。
- 当前题图状态变为“已学习”。

保存后显示明显正反馈：

- 打卡成功。
- 今日已学题数。
- 对应科目地块成长。
- 如触发勋章或抽奖机会，按现有机制提示。

## 8. 数据模型草案

### 8.1 存储原则

图片本体不存 localStorage。图片文件放 IndexedDB，localStorage 只存轻量索引和题目元数据。

原因：

- localStorage 容量小，通常只有 5MB 左右。
- 图片转 base64 后体积大，容易爆容量。
- IndexedDB 更适合存 Blob。

### 8.2 IndexedDB

数据库名建议：

```text
mochi_question_desk
```

对象仓库：

```text
question_images_blob
```

保存：

```javascript
{
  id: "img_...",
  blob: Blob,
  mimeType: "image/png",
  createdAt: ISOString
}
```

### 8.3 localStorage keys

新增：

```text
question_desk_images
question_desk_items
question_desk_ui_state
```

Phase 1 不需要 `question_desk_name_template`，只需要当天递增编号。

### 8.4 question_desk_images

```javascript
[
  {
    id: "img_1718700000000_abcd",
    name: "未分类-拍题-2026-06-18-01",
    shortName: "未分类-拍题-06-18-01",
    subject: "math" | "physics" | "chemistry" | "uncategorized",
    sourceType: "拍题",
    date: "YYYY-MM-DD",
    mimeType: "image/png",
    width: 1280,
    height: 960,
    status: "new" | "asked" | "saved",
    createdAt: ISOString,
    updatedAt: ISOString,
    savedLogId: ""
  }
]
```

### 8.5 question_desk_items

Phase 1 一张图片就是一个题目，因此题目对象和图片一一对应，不存 rect、marker、crop blob。

```javascript
[
  {
    id: "item_1718700000000_abcd",
    imageId: "img_1718700000000_abcd",
    subject: "math" | "physics" | "chemistry" | "uncategorized",
    nodeId: "",
    nodeLabel: "",
    status: "new" | "asked" | "drafted" | "saved",
    title: "未分类-拍题-06-18-01",
    chat: [
      {
        role: "user" | "assistant",
        content: "...",
        createdAt: ISOString
      }
    ],
    recordDraft: {
      subject: "math",
      nodeId: "math_function",
      nodeLabel: "函数",
      questionsCompleted: 1,
      stars: 2,
      painPoint: "...",
      originalQuestion: "...",
      routine: "...",
      date: "YYYY-MM-DD",
      meta: {
        source: "lesson",
        errorType: "审题",
        stuckStep: "",
        keyInsight: "...",
        tags: ["题桌"],
        confidence: 3,
        timeSpentMinutes: 12
      }
    },
    savedLogId: "",
    createdAt: ISOString,
    updatedAt: ISOString
  }
]
```

注意：

- `nodeId` 必须在保存前由 `subject + nodeLabel` 解析得到，不能让 AI 自由生成未知知识点。
- `nodeLabel` 必须落在现有预设知识点列表。
- `meta.source` 使用现有 `normalizeSource()` 能识别的来源词；题桌新学题默认用 `lesson`，如果后续想显示“错题”标签，需要同步扩展来源词表和 UI 文案。

### 8.6 备份与迁移

题桌承诺“问过的题以后能找回来”，因此不能只把图片放 IndexedDB 后完全不管备份。

Phase 1 至少需要一个手动出口：

- 导出题桌图片包：导出 JSON 索引 + 图片 Blob 打包文件，或生成一个浏览器可下载的 zip。
- 导入题桌图片包：恢复 localStorage 索引和 IndexedDB 图片。

如果 Phase 1 暂时不实现 zip，应在 UI 中明确提示：题图保存在本机浏览器，换电脑前请先导出题桌包。

现有 MochiStudy 备份文件的 `version` 和 `data` 主结构不在 Phase 1 修改范围内。后续可评估是否把题桌包并入总备份。

## 9. 分阶段路线

### Phase 0：视觉 AI go/no-go 验证

目标：决定题桌能否成立。

必须验证：

- 用户实际配置的 API endpoint/model 是否支持图片输入。
- 当前 `modules/ai.js` 的 `callAI(systemPrompt, userMessage)` 如何升级为多模态 content 数组。
- 多轮讲题所需 `max_tokens` 是否足够，默认 1000 不够时如何配置。
- 能否把粘贴图片作为 Blob/base64/image_url 传给模型。
- 模型能否读懂真实题目截图，并按高中生讲题方式回复。
- 能否让模型稳定输出可解析的学习记录草稿。

go/no-go：

- Go：用户实际可用模型支持图片输入和多轮问答，进入 Phase 1。
- No-go：先做 AI 配置升级或模型选择，不发布只会把学生带回外部 AI 的题桌。

辅助验证：

- 能监听粘贴事件并读取剪贴板图片。
- 能将图片 Blob 存入 IndexedDB。
- 能从 IndexedDB 读取 Blob 并显示。
- 能调用现有 `applyMochiRecord()` 生成学习卡片。

### Phase 1：一图一题站内 AI MVP

目标：跑通“粘图 -> 站内 AI 问答 -> 生成记录草稿 -> 保存学习档案 -> 成长反馈”的核心闭环。

范围：

- 在“学习”页内新增题桌子页或将题桌作为学习页默认入口，不新增第 5 个底部主导航。
- 题桌清爽三栏布局。
- 支持粘贴图片。
- 支持拖拽/点击上传图片作为兜底。
- 粘贴后自动命名，名称可编辑，当天编号自动递增。
- 左侧基础文件栏：
  - 收件箱
  - 数学
  - 物理
  - 化学
  - 已学习
  - 未学习
- 中间单图查看器。
- 一张图片默认就是一道题。
- 右侧站内 AI 学习面板。
- 支持图片输入 AI。
- 支持多轮对话。
- AI 生成学习记录草稿。
- 保存到学习档案并刷新现有系统。
- 保存成功后显示农场成长、今日题数、sparkle、勋章提示等反馈。
- 提供题桌图片包导出入口，或至少明确本机存储提示。

不做：

- 框选题目区域。
- 裁剪题目截图。
- 题旁小圆标。
- 命名模板反推。
- PDF。
- 跨题排序。

成功标准：

- 学生可以在 30 秒内把一道题图粘进题桌并发起站内 AI 问答。
- 学生不需要离开 MochiStudy。
- 学完后可以保存为现有学习档案中的一张卡片。
- 复习队列和今日学习报告能看到这条记录。
- 保存成功时能看到明确成长反馈。

### Phase 2：一页多题与题旁标记

目标：支持半页/整页卷子里的多道题。

范围：

- 在图片上框选题目区域。
- 根据相对坐标保存 rect。
- 支持裁剪题目截图作为 AI 输入。
- 在原图位置显示题旁小圆标。
- 点击小圆标回到对应题目的对话和记录。

成功标准：

- 一张半页/整页卷子可以创建多道题，并各自保存学习记录。
- 缩放后框选区域和标记不明显错位。

### Phase 3：题本与整理能力

目标：解决长期使用后的资料管理问题。

范围：

- 自定义题本。
- 重命名题图。
- 命名模板记忆和反推。
- 移动题图到题本。
- 按科目、日期、是否已学习、知识点过滤。
- 搜索题图名称和题目标题。
- 批量删除或归档。

成功标准：

- 使用一个月后，学生仍能找回问过的题。

### Phase 4：啃卷子批量排序

目标：恢复 `gaokao啃卷子.md` 最值钱的跨题排优先级能力。

范围：

- 支持导入一批题图。
- 学生选择若干未学题。
- AI 根据高考考频和短期提分空间排序。
- 学生按推荐顺序逐题进入一图一题学习面板。

成功标准：

- 学生面对一批不会的题时，不再平均用力，而是先学最高 ROI 的题。

### Phase 5：PDF 支持

目标：支持正式试卷和电子资料。

范围：

- 上传 PDF。
- 多页预览。
- 每页像图片一样创建题目。
- 题目锚点绑定到 PDF 页码和相对坐标。
- 左侧文件栏支持 PDF 展开页列表。

成功标准：

- 一份 PDF 试卷可以逐页学习并保存学习记录。

### Phase 6：学习橱窗与反向跳转

目标：让题目记录成为长期学习资产入口。

范围：

- 点题旁标记展示题目截图、AI 对话、卡点、套路、保存状态、复习历史。
- 从学习档案卡片反向跳回原题位置。
- 今日学习报告展示题目截图。
- 复习时可以直接打开原题图。

成功标准：

- 学习档案不只是文字卡片，而能回到原始题目现场。

## 10. Agent 分工建议

### Agent A：视觉 AI 可行性

职责：

- 检查并升级 `modules/ai.js` 的多模态调用能力。
- 验证用户实际 endpoint/model 能否吃图片。
- 调整 `max_tokens` 或支持可配置输出长度。
- 验证多轮对话和记录草稿生成。

交付：

- Phase 0 go/no-go 结论。
- 最小多模态调用 API。

### Agent B：数据层

职责：

- IndexedDB 封装。
- 图片 Blob 存取。
- localStorage 索引读写。
- 当天编号规则。
- 题桌图片包导出/导入或本机存储警告。

交付：

- `modules/questionDeskStore.js` 或同类模块。
- 清晰 API：saveImageBlob、readImageBlob、listImages、saveItem、listItems。

## 11. Phase 1 验收标准

功能验收：

- 可以通过粘贴图片创建题图。
- 图片进入左侧文件栏，名称不是乱码。
- 连续粘贴图片时，当天编号自动递增。
- 可以点击左侧文件打开图片。
- 一张图默认就是一道题。
- 可以在右侧面板发起站内 AI 问答。
- AI 能读取题图并进行多轮讲题。
- AI 能生成学习记录草稿。
- 记录草稿的科目、知识点、星级、卡点、原题、套路可确认或编辑。
- 点击保存后，现有学习档案出现对应卡片。
- 农场、复习队列、勋章状态能正常刷新。
- 保存成功后有明显成长反馈。
- 有题桌图片包导出入口，或至少有明确的本机存储和换设备提醒。

体验验收：

- 桌面端三栏布局清晰。
- 主学习区清爽，不像小游戏页面。
- 保存成功反馈有成就感，不削弱现有激励系统。
- 学生不需要理解文件系统也能完成一次学习。
- 学生不需要离开 MochiStudy 去外部 AI。
- 不上传到后端服务器，仍保持本地前端应用特性。

技术验收：

- Phase 0 已证明用户实际 AI 配置支持图片输入。
- `callAI` 或新 AI 调用层支持多模态 content。
- 多轮讲题 token 上限足够，不沿用 1000 的硬限制。
- 图片不写入 localStorage。
- JS 文件通过 `node --check`。
- 不修改 `study_log` 单条记录字段结构。
- 不修改备份文件 `version` 和主结构。
- 不引入后端依赖。

## 12. 风险与待确认问题

### 12.1 AI 图片理解能力

这是 Phase 0 go/no-go，不是 Phase 2 风险。用户实际配置的 endpoint/model 必须支持图片输入，否则题桌的站内学习前提不成立。

### 12.2 剪贴板兼容性

不同浏览器对读取剪贴板图片支持不完全一致。需要保留拖拽和点击上传作为兜底。

### 12.3 IndexedDB 备份

图片存在 IndexedDB，换设备可能丢失原题图。Phase 1 必须至少提供手动导出题桌包或明确提醒；后续再评估并入总备份。

### 12.4 移动端体验

Phase 1 不做框选，但移动端仍要处理粘贴/上传、图片查看和 AI 面板空间不足的问题。

### 12.5 学科自动识别

第一阶段不依赖自动识别。保存记录时由 AI 建议，学生确认，系统校验到现有预设知识点。

### 12.6 啃卷子定位

`gaokao啃卷子.md` 的核心是跨题排序。Phase 1 一图一题更适合使用“逐题讲解/从零带学”的提示词。啃卷子的排序能力放到 Phase 4 批量题图阶段。

## 13. 当前决策

- 新页面命名：题桌。
- 第一版输入方式：以粘贴图片为主，拖拽和上传为兜底。
- 第一版资料形态：左侧文件栏 + 中间单图查看器 + 右侧 AI 学习面板。
- 第一版一图一题，不做框选、裁剪、题旁标记。
- 第一版不做无限画布。
- 第一版不做 PDF。
- 第一版不新增第 5 个移动端底部主导航，先放入“学习”体系。
- 第一版必须接入站内视觉 AI，否则不发布假闭环。
- 第一版必须保留保存成功后的成长反馈。
- 第一版必须考虑题图本机存储的导出或提醒。
- 第一版目标是验证“站内学题闭环”，不追求完整 GoodNotes 级体验。

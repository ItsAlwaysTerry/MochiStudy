# MochiStudy 题桌 PRD

## 0. 方向修订（2026-06-18，Phase 1 已上线后）

Phase 1 MVP 已经按本 PRD 早期版本落地：题桌作为「学习」tab 下的一个子页，嵌在旧 MochiStudy 的游戏化外壳（首页/农场/勋章/赛季侧边栏）里。

产品负责人确认这与目标心智不符，做出方向调整，主从关系对调：

- **题桌是一打开就见的主界面（默认外壳）**，运行在自己干净、低干扰的框里，不再寄居于旧网站的游戏化侧边栏，也不再是「学习」tab 的子页。
- **整套旧 MochiStudy（首页/农场、勋章、赛季、复习队列、学习档案、今日报告、设置）降级为一个「世界」**，从题桌角落一个低调按钮点进去、全屏打开；旧网站里要有明显的「返回题桌」入口。
- 题桌只负责「传题 + 学新题」这条当下学习主线；复习队列 / 学习档案 / 今日报告等「回顾与记录」功能**全部留在旧网站「学习」tab 不动**。职责切分：题桌 = 当下学，旧网站 = 回顾与成长。

本次修订只翻主从结构，不要求重做题桌三栏内部视觉，也不提前做框选 / 浮窗（仍属 Phase 2/6）。下文第 2、6.1、6.2、13 节及 Phase 1 范围已据此更新；如有残留旧表述与本节冲突，以本节为准。

实现交接清单：

### 0.1 题桌形态反馈与后续两轮（2026-06-18）

外壳重构上线后，产品负责人对题桌内部形态提出反馈，目标心智是「像笔记软件：题目在中间，平时只看题；AI 问答按需出现，可收起也可展开到全屏；问过的题带小标，点开看对应的学习档案知识卡片」。当前 Phase 1 实现是「固定画布 + 常驻较窄的 AI 侧边栏 + 偏程序员风的密集表单」，与该心智有差距。

据此排两个独立的后续轮次，分开实现：

- **Round A = Phase 1.5 题桌清爽化**（先做，见效快）：AI 面板可收起/展开到全屏、学习记录草稿表单从「程序员表单」重做成清新卡片、整体视觉按原则 4.5 打磨。不碰框选 / 小标 / 浮窗。
- **Round B = Phase 2 增强**（后做）：题旁小标 + 浮窗，浮窗内含 AI 对话与该题关联知识点的「学习档案知识卡片」。

注意：这两轮只调整题桌内部体验，不推翻已落地的外壳结构、数据模型、保存闭环。详见第 9 节 Phase 1.5、Phase 2。

## 1. 背景

MochiStudy 目前更像一个“学完之后的记录系统”：学生在外部 AI 或其他学习场景里完成一道题，再把 AI 输出的 MOCHI-RECORD 粘回网站，网站负责学习档案、复习队列、农场、勋章和今日报告。

真实摩擦不是“缺一个上传题图的地方”，而是学习现场不在 MochiStudy 里。学生需要在题图、外部 AI、MochiStudy 导入框之间来回切换，操作链路长，记录也依赖学生主动复制粘贴。

题桌要解决的是：学生看到题不会做的那一刻，可以直接把题图粘进 MochiStudy，在站内问 AI，问完后把学习结果保存进现有学习档案。

第一版必须上线当天就比现状更省事。不能先做一套很重的图片管理和框选系统，却仍然要求学生去外部 AI 复制粘贴。站内视觉 AI 对话是题桌成立的前提，必须前置验证并进入第一版闭环。

## 2. 产品定位

题桌是 MochiStudy 的主学习工作台，也是用户打开应用后默认落地的主界面（应用外壳）。旧的游戏化 MochiStudy（首页/农场、勋章、赛季等）不再是顶层框架，而是题桌之外、通过一个按钮进入的次级世界（见 0、6.1）。

第一阶段题桌内部不做无限白板，也不是 GoodNotes 级批注工具（框选 / 浮窗属 Phase 2/6），它的主界面形态是：

- 左侧：题图文件栏，用来管理粘贴进来的题目图片。
- 中间：当前题图查看器，一张图默认就是一道题。
- 右侧：站内 AI 学习面板，承载题图、对话、学习记录草稿、保存到学习档案。

一句话定义：

> 学生把一道题图粘进题桌，在站内让 AI 带着学，学完自动变成 MochiStudy 的学习卡片。

后续阶段再扩展为“一页卷子框多题”“题旁标记”“PDF 试卷”“从学习档案反跳回原题现场”。

## 3. 目标用户与场景

目标用户是基础薄弱、需要高频陪伴式讲题的高中生。

真实使用场景：

- 学生做练习册，遇到不会的题，直接截图或拍照粘进题桌。
- 学生日常更多是零散上传一道题，而不是先整理完整试卷。
- 学生偶尔会拍半页卷子或一页卷子，但第一版仍建议让他先按“一图一题”的方式学习。
- 学生后期可能向老师要 PDF 版试卷，再导入题桌逐题学习。
- 学生不想先整理资料，但希望问过的题以后能找回来。
- 家长或老师希望看到学生今天到底搞懂了哪些题、哪些知识点。

## 4. 产品原则

### 4.1 第一版必须比旧流程短

旧流程是：

```text
截图 -> 外部 AI -> 复制 MOCHI-RECORD -> 粘进导入框
```

题桌 Phase 1 必须变成：

```text
粘图 -> 站内 AI 问答 -> 确认保存
```

如果第一版仍要求“粘图 -> 去外部 AI -> 复制输出 -> 粘回题桌”，它就不是 MVP，而是比现状更累的中间态。

### 4.2 一图一题先跑通

第一版不做框选、裁剪和题旁标记。一张图片默认就是一道题。这样更符合学生日常“遇到一道题就拍一道题”的真实输入方式，也能把工程精力集中在站内 AI 和保存闭环上。

### 4.3 先学习，后整理

题图先进入收件箱，不能要求学生上传时就完整填写科目、知识点、标签。整理应该发生在保存学习记录时，由 AI 输出和学生确认共同完成。

### 4.4 题目是最小单位

系统核心对象不是“笔记”或“试卷”，而是一道被问过、被讲过、被保存的题目。

### 4.5 清爽界面，但保留成长反馈

题桌的学习区要清爽、成熟、低干扰：浅灰背景、白色纸面、轻工具栏、少量蓝色强调。

但保存成功那一刻必须保留 MochiStudy 的动力引擎：农场成长、sparkle、连续学习、勋章或今日进度反馈。题桌不能把“学完看得见成长”的闭环藏起来。

### 4.6 记录自动化

学生不应该再手动复制 MOCHI-RECORD。AI 对话结束后，系统生成学习记录草稿，学生确认即可保存到现有 `study_log` 和 `study_card_meta`。

## 5. 非目标

Phase 1 不做以下内容：

- PDF 导入和多页 PDF 渲染。
- OCR 自动识别题目边界。
- 半页/整页卷子的框选、裁剪、坐标标记。
- 题旁小圆标和从原图位置回看对话。
- 手写批注、画笔、橡皮、完整 GoodNotes 级标注系统。
- 复杂自定义题本、全文搜索、批量整理。
- 从用户手动改名中反推命名模板。
- 跨题“啃卷子”优先级排序。
- 修改现有 `study_log` 字段结构。
- 推翻现有农场、学习档案、复习队列、勋章系统。

这些是后续阶段的方向，不属于第一轮验证。

## 6. 信息架构

### 6.1 主导航策略（2026-06-18 修订）

题桌是默认主界面，不再嵌在旧网站的游戏化导航里。

两个世界，主从分明：

- **题桌世界（默认）**：应用打开即落在题桌，运行在自己干净的外壳里——不显示旧的游戏化侧边栏（首页/学习/勋章/赛季/设置）和移动端底部导航。视觉低干扰，焦点在「传题 + 站内 AI 学题」。
- **旧 MochiStudy 世界（次级）**：从题桌角落一个低调按钮（例如「我的成长 / 档案」）点进去，全屏打开旧网站，连同它原有的侧边栏和全部页面（首页/农场、学习、勋章、赛季、设置）。旧网站里必须有明显的「返回题桌」入口，随时回到主界面。

要点：

- 不在旧网站里再给题桌留一个子 tab——题桌已经是外壳，不能同时又是「学习」tab 的子页。原「学习」tab 恢复为 今日 / 复习 / 档案。
- 复习队列 / 学习档案 / 今日报告 留在旧网站「学习」tab 不动（见 0）。
- 不新增第 5 个底部主导航；这一点仍成立，但实现方式从「塞进学习子页」改为「题桌作外壳、旧网站整体收到一个按钮后面」。

推荐实现方式（供实现 Agent 参考，非强制）：

- 默认路由从 `home` 改为 `desk`，`#desk` 升级为顶层独立路由，全屏渲染题桌，不再经过 `renderLearn` 的子 tab。
- 用 `body.desk-mode` 之类的 body class 切换外壳：题桌路由时隐藏 `.side-nav` / `.bottom-nav` / 旧顶栏；进入旧网站路由时恢复。
- 题桌内放「进旧网站」按钮 → `navigate("home")`；旧网站侧边栏顶部 + 顶栏放「← 题桌」按钮 → `navigate("desk")`。
- 从「学习」tab 栏移除「题桌」，默认子 tab 回到「复习」。

### 6.2 Phase 1 页面结构

以下三栏是题桌自己的全屏外壳内部，不再外包旧网站的游戏化侧边栏。顶部轻工具栏需含一个低调的「进旧网站 / 我的成长」入口。

桌面端：

```text
+------------------------------------------------------+
| 顶部轻工具栏：粘贴题图 / 上传兜底 / 导出题桌包 / 我的成长 |
+-------------+----------------------+-----------------+
| 左侧文件栏   | 中间题图查看器         | 右侧 AI 学习面板 |
| 收件箱       | 当前题图              | 题目截图         |
| 数学         | 一图一题              | 多轮对话         |
| 物理         |                      | 记录草稿         |
| 化学         |                      | 保存到学习档案   |
| 已学习       |                      | 成长反馈         |
| 未学习       |                      |                 |
+-------------+----------------------+-----------------+
```

移动端：

- 默认显示题图和 AI 学习入口。
- 左侧文件栏变为抽屉。
- 右侧学习面板变为底部弹层或全屏面板。

## 7. 核心用户路径

### 7.1 粘贴题图

1. 学生复制一张图片。
2. 回到题桌，按 `Ctrl+V` 或点击“粘贴题图”。
3. 系统读取剪贴板图片。
4. 弹出轻量命名条，默认名称已生成并全选。
5. 学生直接回车保存，或输入新名称覆盖。
6. 图片进入左侧文件栏，并在中间打开。
7. 右侧 AI 学习面板自动准备好这张题图。

### 7.2 Phase 1 自动命名

默认命名格式：

```text
未分类-拍题-YYYY-MM-DD-编号
```

例子：

```text
未分类-拍题-2026-06-18-01
未分类-拍题-2026-06-18-02
```

Phase 1 命名规则：

- 默认科目为“未分类”。
- 默认类型为“拍题”。
- 同一天连续粘贴时，编号自动递增。
- 命名输入框默认全选，用户直接打字即可替换。
- 用户可以手动选择科目，但系统不从自定义名称里反推模板。
- 左侧文件栏显示短名称，例如 `未分类-拍题-06-18-02`。

后续整理阶段再支持“数学-周测-日期-编号”等模板记忆和反推。

### 7.3 站内 AI 学习

1. 学生打开一张题图。
2. 右侧面板显示题图缩略图和输入框。
3. 学生输入“这题怎么做”“我看不懂第一步”等问题。
4. 系统把题图和学生问题发给支持图片输入的 AI。
5. AI 按“题桌逐题讲解”规则引导学生，不直接只给答案。
6. 支持多轮追问。
7. 对话结束后，学生点击“生成学习记录”或 AI 自动生成记录草稿。

Phase 1 不能依赖外部 AI 复制粘贴。如果 Phase 0 证明当前 API 配置不能处理图片输入，题桌进入 no-go 或先做 API 配置升级，不发布假闭环。

Phase 1 需要新增或明确一份单题视觉讲解 prompt，例如 `skill/gaokao题桌.md`。它不同于 `gaokao啃卷子.md`：后者核心是跨题排序，放到 Phase 4；题桌第一版需要的是“看一张题图，先问卡点，再逐步提示，最后生成学习记录草稿”的单题讲解人格。

### 7.4 保存到学习档案

题目面板生成学习记录草稿后，学生点击“保存到学习档案”。

保存前必须确认：

- 科目必须是数学 / 物理 / 化学之一。
- 知识点必须来自现有预设列表。
- 星级为 1-3。
- 原题描述、卡点、今日套路有合理内容。

保存时：

- 将 `nodeLabel` 解析成现有 `nodeId`。
- 调用现有 `window.MochiApp.applyMochiRecord(record)`。
- 写入现有 `study_log`。
- 写入现有 `study_card_meta`。
- 保存 `savedLogId` 回题桌题目记录。
- 刷新学习档案、复习队列、农场、勋章、今日学习报告。
- 当前题图状态变为“已学习”。

保存后显示明显正反馈：

- 打卡成功。
- 今日已学题数。
- 对应科目地块成长。
- 如触发勋章或抽奖机会，按现有机制提示。

## 8. 数据模型草案

### 8.1 存储原则

图片本体不存 localStorage。图片文件放 IndexedDB，localStorage 只存轻量索引和题目元数据。

原因：

- localStorage 容量小，通常只有 5MB 左右。
- 图片转 base64 后体积大，容易爆容量。
- IndexedDB 更适合存 Blob。

### 8.2 IndexedDB

数据库名建议：

```text
mochi_question_desk
```

对象仓库：

```text
question_images_blob
```

保存：

```javascript
{
  id: "img_...",
  blob: Blob,
  mimeType: "image/png",
  createdAt: ISOString
}
```

### 8.3 localStorage keys

新增：

```text
question_desk_images
question_desk_items
question_desk_ui_state
```

Phase 1 不需要 `question_desk_name_template`，只需要当天递增编号。

Phase 3 轻量化后不新增显性题本 key。题图整理主要依靠 `name`、`subject` 和搜索；历史试验产生的 `question_desk_notebooks` 只作为清理兼容 key，不作为当前产品主模型。

### 8.4 question_desk_images

```javascript
[
  {
    id: "img_1718700000000_abcd",
    name: "未分类-拍题-2026-06-18-01",
    shortName: "未分类-拍题-06-18-01",
    subject: "math" | "physics" | "chemistry" | "uncategorized",
    sourceType: "拍题",
    date: "YYYY-MM-DD",
    mimeType: "image/png",
    width: 1280,
    height: 960,
    status: "new" | "asked" | "saved",
    createdAt: ISOString,
    updatedAt: ISOString,
    savedLogId: ""
  }
]
```

### 8.5 question_desk_items

Phase 1 一张图片就是一个题目，因此题目对象和图片一一对应，不存 rect、marker、crop blob。

```javascript
[
  {
    id: "item_1718700000000_abcd",
    imageId: "img_1718700000000_abcd",
    subject: "math" | "physics" | "chemistry" | "uncategorized",
    nodeId: "",
    nodeLabel: "",
    status: "new" | "asked" | "drafted" | "saved",
    title: "未分类-拍题-06-18-01",
    chat: [
      {
        role: "user" | "assistant",
        content: "...",
        createdAt: ISOString
      }
    ],
    recordDraft: {
      subject: "math",
      nodeId: "math_function",
      nodeLabel: "函数",
      questionsCompleted: 1,
      stars: 2,
      painPoint: "...",
      originalQuestion: "...",
      routine: "...",
      date: "YYYY-MM-DD",
      meta: {
        source: "lesson",
        errorType: "审题",
        stuckStep: "",
        keyInsight: "...",
        tags: ["题桌"],
        confidence: 3,
        timeSpentMinutes: 12
      }
    },
    savedLogId: "",
    createdAt: ISOString,
    updatedAt: ISOString
  }
]
```

注意：

- `nodeId` 必须在保存前由 `subject + nodeLabel` 解析得到，不能让 AI 自由生成未知知识点。
- `nodeId` 必须来自现有节点表或解析 helper，例如当前 `函数` 对应 `math_function`；实现者不能手写或猜测 id 格式。
- `nodeLabel` 必须落在现有预设知识点列表。
- `questionsCompleted` 在现有保存逻辑里固定为 1，草稿里保留只是为了表达“一题一记录”，不是新的可配置输入。
- `meta.source` 使用现有 `normalizeSource()` 能识别的来源词；题桌新学题默认用 `lesson`，如果后续想显示“错题”标签，需要同步扩展来源词表和 UI 文案。

### 8.6 Phase 1 学习记录草稿契约

Phase 1 不再依赖 AI 输出 `---MOCHI-RECORD-START---` 文本块，但必须锁定一份可解析的草稿格式，避免 prompt 和 parser 漂移。

题桌单题 prompt 必须输出以下中文标签，Phase 1 parser 只承认这些字段：

```text
【MochiStudy 学习记录草稿】
科目：
知识点：
掌握星级：
卡点记录：
原题：
今日套路：
错误类型：
卡住步骤：
关键突破：
题型标签：
信心分：
耗时分钟：
学习日期：
```

字段映射：

```text
科目 -> recordDraft.subject
知识点 -> recordDraft.nodeLabel，并解析为 recordDraft.nodeId
掌握星级 -> recordDraft.stars
卡点记录 -> recordDraft.painPoint
原题 -> recordDraft.originalQuestion
今日套路 -> recordDraft.routine
错误类型 -> recordDraft.meta.errorType
卡住步骤 -> recordDraft.meta.stuckStep
关键突破 -> recordDraft.meta.keyInsight
题型标签 -> recordDraft.meta.tags
信心分 -> recordDraft.meta.confidence
耗时分钟 -> recordDraft.meta.timeSpentMinutes
学习日期 -> recordDraft.date
```

`recordDraft.meta.source` 不从 AI 草稿解析，由题桌保存代码固定注入 `lesson`。题桌 Phase 1 是站内新学场景，不应该让 AI 猜“学习来源”；如果后续要区分“错题/新学/复习”，必须先扩展来源词表和 UI，而不是让 prompt 自由输出。

字段值解析规则：

- `科目`：只接受数学 / 物理 / 化学，并映射为 `math` / `physics` / `chemistry`。无法识别时必须让学生手动选择。
- `知识点`：必须精确命中该科目的预设知识点列表，才能写入 `nodeLabel` 并解析 `nodeId`。如果 AI 输出“二次函数”“受力分析”等非预设名称，必须弹给学生手动选择最接近的预设知识点，绝不允许使用 `nodeIdFromLabel()` 的默认 fallback 静默落到该科目第一个知识点。
- `掌握星级`：解析为 1-3；需要容忍 `2`、`2星`、`★★☆` 等格式，无法解析时让学生选择。
- `信心分`：解析为 1-5；需要容忍 `3`、`3分` 等格式，无法解析时可置 0 或让学生选择。
- `耗时分钟`：解析为整数分钟；无法解析时可置 0。
- `题型标签`：按 `、`、`,`、`，` 或换行切分为数组，去掉空项。
- 文本字段：去掉首尾空白，保留必要换行；空值保存前必须在 UI 中提醒学生补齐关键字段。

`原题` 是未来复习和 AI 分析会依赖的 durable text，不能默认写“见题桌原图”。AI 必须尽力转写题干核心文字、数字和关键公式；只有公式/图形过密无法完整转写时，才可以在文字摘要后补“详见题桌原图”。因为 IndexedDB 图片在换设备时可能丢失，`originalQuestion` 不能只是一个指向本机图片的悬空引用。

### 8.7 备份与迁移

题桌承诺“问过的题以后能找回来”，因此不能只把图片放 IndexedDB 后完全不管备份。

Phase 1 至少需要一个手动出口：

- 导出题桌图片包：导出 JSON 索引 + 图片 Blob 打包文件，或生成一个浏览器可下载的 zip。
- 导入题桌图片包：恢复 localStorage 索引和 IndexedDB 图片。

如果 Phase 1 暂时不实现 zip，应在 UI 中明确提示：题图保存在本机浏览器，换电脑前请先导出题桌包。

现有 MochiStudy 备份文件的 `version` 和 `data` 主结构不在 Phase 1 修改范围内。后续可评估是否把题桌包并入总备份。

## 9. 分阶段路线

### Phase 0：视觉 AI go/no-go 验证

目标：决定题桌能否成立。

必须验证：

- 用户实际配置的 API endpoint/model 是否支持图片输入。
- 当前 `modules/ai.js` 的 `callAI(systemPrompt, userMessage)` 如何升级为多模态 content 数组。
- 多轮讲题所需 `max_tokens` 是否足够，默认 1000 不够时如何配置。
- 能否把粘贴图片作为 Blob/base64/image_url 传给模型。
- 模型能否读懂真实题目截图，并按高中生讲题方式回复。
- 能否让模型稳定输出可解析的学习记录草稿。

go/no-go：

- Go：用户实际可用模型支持图片输入和多轮问答，进入 Phase 1。
- No-go：先做 AI 配置升级或模型选择，不发布只会把学生带回外部 AI 的题桌。

辅助验证：

- 能监听粘贴事件并读取剪贴板图片。
- 能将图片 Blob 存入 IndexedDB。
- 能从 IndexedDB 读取 Blob 并显示。
- 能调用现有 `applyMochiRecord()` 生成学习卡片。

### Phase 1：一图一题站内 AI MVP

目标：跑通“粘图 -> 站内 AI 问答 -> 生成记录草稿 -> 保存学习档案 -> 成长反馈”的核心闭环。

范围：

- 题桌作为应用默认主界面（独立外壳），不嵌入旧网站游戏化侧边栏，不作为「学习」tab 子页；旧 MochiStudy 经题桌内一个按钮全屏进入，并提供「返回题桌」入口（见 0、6.1）。原 Phase 1 已落地的「学习子页」形态按本修订重构。
- 题桌清爽三栏布局。
- 支持粘贴图片。
- 支持拖拽/点击上传图片作为兜底。
- 粘贴后自动命名，名称可编辑，当天编号自动递增。
- 左侧基础文件栏：
  - 收件箱
  - 数学
  - 物理
  - 化学
  - 已学习
  - 未学习
- 中间单图查看器。
- 一张图片默认就是一道题。
- 右侧站内 AI 学习面板。
- 支持图片输入 AI。
- 支持多轮对话。
- 新增 `skill/gaokao题桌.md` 或等价的单题视觉讲解 prompt。
- AI 生成学习记录草稿。
- 保存到学习档案并刷新现有系统。
- 保存成功后显示农场成长、今日题数、sparkle、勋章提示等反馈。
- 提供题桌图片包导出入口，或至少明确本机存储提示。

不做：

- 框选题目区域。
- 裁剪题目截图。
- 题旁小圆标。
- 命名模板反推。
- PDF。
- 跨题排序。

成功标准：

- 学生可以在 30 秒内把一道题图粘进题桌并发起站内 AI 问答。
- 学生不需要离开 MochiStudy。
- 学完后可以保存为现有学习档案中的一张卡片。
- 复习队列和今日学习报告能看到这条记录。
- 保存成功时能看到明确成长反馈。

### Phase 1.5：题桌清爽化（Round A，下一轮之一，2026-06-18）

目标：把 Phase 1 已跑通的闭环，从「固定画布 + 常驻窄侧栏 + 程序员表单」打磨成接近笔记软件的清爽体验。只改题桌内部交互与视觉，不改数据模型、保存闭环、外壳结构，不做框选 / 小标 / 浮窗（留 Round B）。

范围：

- **AI 面板可收起 / 展开**：
  - 收起：中间题图区铺满，平时「只看题」。
  - 展开：AI 面板可放大（更宽，或覆盖式大面板/全屏），方便长对话和读草稿。
  - 提供明显但低调的开合控件；记住上次开合状态（可存 `question_desk_ui_state`）。
- **学习记录草稿表单重做成清新卡片**：
  - 关键字段突出：卡点记录、原题、今日套路、星级。
  - 次要 meta 字段（错误类型、卡住步骤、关键突破、题型标签、信心分、耗时分钟）默认折叠或弱化，渐进展开，不一上来糊一屏下拉框。
  - 尽量用轻量控件替代密集 `<select>`：如星级用星星点选、科目/常用值用 chip 或分段控件；知识点仍须从预设列表精确选择（保留校验，见 7.4、8.6），但呈现要柔和。
  - 字段标签、留白、圆角、配色按原则 4.5（浅灰背景、白纸面、少量蓝色强调）统一，去掉硬核边框和后台管理页观感。
- **整体视觉打磨**：三栏配色、间距、按钮、空状态、加载态统一到清新风；移动端保持可用。

不做：

- 框选、裁剪、题旁小标、浮窗、学习档案知识卡片（全部留 Round B / Phase 2）。
- 改 `study_log`、`study_card_meta`、备份结构、保存逻辑、知识点校验规则。

成功标准：

- 学生能一键收起 AI 面板，中间只剩题图；也能把面板展开到足够大读对话和草稿。
- 生成草稿后看到的是友好的卡片，关键字段一眼可改，次要字段默认收着，不像后台表单。
- 整体观感清新、低干扰，符合原则 4.5，不再有「很程序员」的硬核感。
- 不引入数据结构或保存闭环回归。

### Phase 2：一页多题、题旁标记与知识卡片浮窗（Round B，下一轮之二）

目标：实现产品负责人的核心愿景——题目带小标、点开浮窗看对话与学习档案知识卡片；同时支持半页/整页卷子里的多道题。本阶段把原 Phase 6「从学习档案反跳回原题 / 知识卡片」的能力前移进浮窗。

范围：

- 在图片上框选题目区域，按相对坐标保存 rect；支持裁剪题目截图作为 AI 输入。
- **题旁小标**：问过 / 已学的题目，在题面对应位置（或单题模式下题图角标、文件项角标）显示小标点，标示「这题问过 / 已学」。
- **点小标弹浮窗**（浮窗或可展开侧栏，二选一由实现定，须延续 Phase 1.5 的清爽风）：
  - 上半部：该题的 AI 对话历史。
  - 下半部：该题关联知识点的「学习档案知识卡片」——复用 `knowledgeMap` 学习卡片的数据（卡点、今日套路、星级、复习历史等），做成小卡片，不必跳走旧网站即可在题桌就地看到。
- 平时可收起所有面板，画布只剩题目，靠小标进入对话与知识卡片。

成功标准：

- 一张半页/整页卷子可以创建多道题，并各自保存学习记录。
- 缩放后框选区域和标记不明显错位。
- 问过的题带小标；点小标能就地看到该题的 AI 对话和对应知识点的学习档案卡片浮窗。

### Phase 3：轻量题盒与找回能力

目标：解决长期使用后的资料管理问题。

范围：

- 左侧只保留低认知负担入口：全部、未整理、数学、物理、化学、搜索。
- 重命名题图；文件名承担“题本感”，例如 `数学-第3周周测-01`。
- 新粘贴题图先进入未整理；AI 识别、生成草稿或保存记录确认科目后，自动归到对应科目。
- 搜索题图名称、知识点、题干转写、识别摘要和草稿原题。
- 单张题图删除；删除只清理题桌图片和对话，不删除已经保存到学习档案的卡片。
- 点空分类时，中间必须显示该分类空态，不能继续显示上一张题。

不做：

- 自定义题本、题本移动、归档、批量归档、显眼的批量删除、知识点筛选。它们会把题桌变成文件系统，增加学生学习成本。

成功标准：

- 使用一个月后，学生仍能找回问过的题。
- 学生不需要理解“题本/归档/批量管理”等概念，只需要粘题、改名、按科目或搜索找题。

### Phase 4：啃卷子批量排序

目标：恢复 `gaokao啃卷子.md` 最值钱的跨题排优先级能力。

范围：

- 支持导入一批题图。
- 学生选择若干未学题。
- AI 根据高考考频和短期提分空间排序。
- 学生按推荐顺序逐题进入一图一题学习面板。

实现口径：

- 不新增批量文件管理系统，不引入题本/归档概念。
- 在题桌里提供「啃卷子」入口，学生先选择要处理的卷子/题图。
- 点击「识别题目」后，AI 逐张素材拆出题目清单；已有手动套索区域的素材优先复用套索题目，未套索的整张卷子调用视觉模型拆题。
- 拆题结果必须尽量保留原卷子的题号；识别不到原题号时显示「未标号」，不能重新按系统生成的序号冒充原题号。
- 默认把识别出的题目视为“不会/要排序”，学生只需要取消已经会的题；排序单位是题目，不是整张卷子。
- 浏览器把选中的题目候选合成一张 A/B/C 编号总览图，继续复用单图视觉 AI 调用。
- 排序结果保存在 `question_desk_ui_state.grindPlan`，只作为学习顺序建议，不写入 `study_log`。

成功标准：

- 学生面对一批不会的题时，不再平均用力，而是先学最高 ROI 的题。

### Phase 5：PDF 支持

目标：支持正式试卷和电子资料。

范围：

- 上传 PDF。
- 多页预览。
- 每页像图片一样创建题目。
- 题目锚点绑定到 PDF 页码和相对坐标。
- 左侧文件栏支持 PDF 展开页列表。

成功标准：

- 一份 PDF 试卷可以逐页学习并保存学习记录。

### Phase 6：学习橱窗与反向跳转

目标：让题目记录成为长期学习资产入口。

注：「点题旁标记看对话 + 学习档案知识卡片」已前移到 Phase 2（见上）。本阶段聚焦反向通路与跨页面打通，与 Phase 2 浮窗不重复。

范围：

- 从旧网站学习档案卡片反向跳回题桌对应原题位置。
- 今日学习报告展示题目截图。
- 复习时可以直接打开原题图。
- 题旁浮窗的深化（保存状态、复习历史等更完整的学习橱窗信息），在 Phase 2 基础上扩展。

成功标准：

- 学习档案不只是文字卡片，而能回到原始题目现场。

## 12. 风险与待确认问题

### 12.1 AI 图片理解能力

这是 Phase 0 go/no-go，不是 Phase 2 风险。用户实际配置的 endpoint/model 必须支持图片输入，否则题桌的站内学习前提不成立。

### 12.2 剪贴板兼容性

不同浏览器对读取剪贴板图片支持不完全一致。需要保留拖拽和点击上传作为兜底。

### 12.3 IndexedDB 备份

图片存在 IndexedDB，换设备可能丢失原题图。Phase 1 必须至少提供手动导出题桌包或明确提醒；后续再评估并入总备份。

### 12.4 移动端体验

Phase 1 不做框选，但移动端仍要处理粘贴/上传、图片查看和 AI 面板空间不足的问题。

### 12.5 学科自动识别

第一阶段不依赖自动识别。保存记录时由 AI 建议，学生确认，系统校验到现有预设知识点。

### 12.6 啃卷子定位

`gaokao啃卷子.md` 的核心是跨题排序。Phase 1 一图一题更适合使用“逐题讲解/从零带学”的提示词。啃卷子的排序能力放到 Phase 4 批量题图阶段。

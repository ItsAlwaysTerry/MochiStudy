# 暑假能量奖励：仅抽奖现金与奖池预览设计

日期：2026-07-21

## 背景

当前奖励经济除了小奖和大奖抽奖，还会在完成 3/4 个视频任务时自动发放 ¥5/¥8，并在每 5 个达标日时固定发放 ¥15。学生端没有清楚展示这两类固定现金，但它们会计入 `summer_reward.paidToday` 和 `weekPaid`。

这会造成两个问题：

1. 学生只看到自己抽中 ¥20，却可能因为隐藏的自动奖励和阶段固定奖励而显示“今日已达上限 ¥40”。
2. 奖励记录只展示抽奖历史，页面累计金额与内部预算计数不一致，无法准确对账。

本次改为“现金只来自抽奖”。完成任务仍负责发券和推进阶段，但不再直接发钱。

## 目标

- 删除完成任务产生的自动 ¥5/¥8 现金。
- 删除每 5 个达标日产生的固定 ¥15 现金。
- 保留小奖券、大奖券、达标日、连续奖励券和阶段进度。
- 每日 ¥40 只限制小奖抽奖；大奖不受每日限额。
- 小奖和大奖共同受每周 ¥150 限额。
- 删除旧状态中已累计的固定现金，只保留真实抽奖金额。
- 在能量浮窗的抽奖按钮中直接预览各奖池金额，不展示概率。

## 非目标

- 不修改小奖和大奖的金额档位或权重。
- 不修改获得小奖券、大奖券、达标日和连续奖励券的条件。
- 不修改 `summer_reward` localStorage key，不重置用户的券或阶段进度。
- 不伪造无法从现有数据可靠还原的旧现金明细。

## 奖励规则

### 任务与发券

- 当天完成 3 个完整视频任务：获得 1 张小奖券，并记为 1 个达标日，不直接发现金。
- 当天完成更多任务：不重复发当日基础小奖券，不直接发现金。
- 连续达标触发的额外小奖券保持现状。
- 每累计 5 个达标日：获得 1 张大奖券，不直接发阶段固定现金。

### 奖池

- 小奖：¥2、¥5、¥10、¥20。
- 大奖：¥20、¥50、¥100。
- 学生端只展示金额档位，不展示权重或概率。

### 预算

- 每日 ¥40：只统计当天小奖抽奖的实发金额。
- 大奖抽奖不占每日 ¥40。
- 每周 ¥150：统计当周所有小奖和大奖抽奖的实发金额。
- 当小奖只剩部分日额度或抽奖只剩部分周额度时，继续沿用现有实发裁剪行为。
- 达到日限额只禁用小奖；达到周限额同时禁用小奖和大奖，且不消耗券。

## 数据模型与迁移

继续使用 `summer_reward`，增加以下字段：

- `rewardModelVersion: 2`：标记已经迁移到“仅抽奖现金”模型。
- `dailyDrawPaidDate: "YYYY-MM-DD"`：小奖日额度对应日期。
- `dailyDrawPaid: number`：该日期小奖抽奖实发合计。

保留字段：

- `paidTodayDate` / `paidToday`：当天全部真实抽奖实发金额，用于“今日到账”显示。
- `weekKey` / `weekPaid`：当周全部真实抽奖实发金额。
- `dailyTickets` / `stageTickets`、`qualDays`、`stages`、`streak`、`history` 等进度字段。

废弃字段：

- `dailySmallDate`。
- `dailySmallPaid`。

首次读取旧模型时进行一次幂等迁移：

1. 只读取 `history` 中的真实抽奖记录。
2. 按 `paidTodayDate` 重算 `paidToday`。
3. 按 `weekKey` 重算 `weekPaid`。
4. 按 `paidTodayDate` 且 `kind === "daily"` 重算 `dailyDrawPaid`。
5. 把 `dailySmallDate` 清空、`dailySmallPaid` 归零。
6. 写入 `rewardModelVersion: 2`，防止重复迁移。

迁移不改变券、达标日、阶段和抽奖历史。由于旧版本没有保存每一天的自动现金明细，奖励记录继续以现有真实抽奖历史为准；这正好符合“以前固定奖励全部删除”的口径。

## 逻辑修改

### `syncEconomy()`

- 删除 `econTierSmall()` 调用和自动小额补差逻辑。
- 删除阶段达成时 `pay(ECON.stageFixed, true)`。
- 保留历史达标日回填、当天达标、连续加券和阶段加券逻辑。
- 删除只服务固定现金的 `pay()` 辅助函数。

### `econDraw(kind)`

- 小奖抽奖用 `dailyDrawPaid` 判断每日剩余额度。
- 小奖实发后同时累加 `dailyDrawPaid`、`paidToday` 和 `weekPaid`。
- 大奖实发后只累加 `paidToday` 和 `weekPaid`，不累加 `dailyDrawPaid`。
- 所有抽奖继续写入 `history`。

### 封顶判断

- `econKindBlocked("daily")` 根据 `dailyDrawPaid` 和 `weekPaid` 判断。
- `econKindBlocked("stage")` 只根据 `weekPaid` 判断。
- 浮窗的 `dailyDrawable` 使用 `dailyDrawPaid`，不再使用全部今日到账。

## 浮窗界面

采用“金额直接写进抽奖按钮”的方案。

- 小奖按钮显示：`¥2 / ¥5 / ¥10 / ¥20`。
- 大奖按钮显示：`¥20 / ¥50 / ¥100`。
- 两个按钮始终显示；无券、日限或周限时禁用，但奖池金额仍可见。
- 按钮同时显示当前券数；无券时显示“暂无券”。
- 头部原来的 `¥今日金额/40` 改成“今日到账 ¥X”，避免把总到账误认为小奖日额度。
- 规则文案删除自动 ¥5/¥8 和阶段固定 ¥15 的暗示，只说明完成任务获得小奖券、累计达标日获得大奖券。
- 若抽奖金额被裁剪，结果文案区分“今日小奖额度已满”和“本周额度已满”。

## 奖励记录页

- 继续只展示 `history` 中的真实抽奖记录。
- “累计”继续按抽奖实发金额求和。
- 页面说明改为现金只来自右下角能量浮窗的抽奖，避免用户误以为还有隐藏奖励。

## 兼容与异常处理

- 旧状态缺少新增字段时自动迁移，不要求用户清空 localStorage。
- `history` 缺失或损坏时按空数组处理；迁移后的预算金额归零，不恢复无法证明的固定奖励。
- 跨日时重置 `paidToday` 和 `dailyDrawPaid`；跨周时重置 `weekPaid`。
- 达到额度时不消耗对应抽奖券。
- 进行中的骰子动画与已确定的抽奖结果保持现有恢复机制。

## 验收场景

1. 旧截图状态：历史当天只有一笔小奖 ¥20，但旧 `paidToday` 为 ¥40。迁移后 `paidToday = 20`、`dailyDrawPaid = 20`，小奖仍可抽，剩余日额度为 ¥20。
2. 完成第 3 个视频任务：只增加 1 张小奖券和 1 个达标日，今日/本周金额不变。
3. 完成第 5 个达标日：只增加 1 张大奖券，今日/本周金额不变。
4. 当天小奖实发累计 ¥40：小奖禁用并保留券；若周额度未满，大奖仍可抽。
5. 当天已经抽取大奖后：今日到账增加，但小奖日额度不变。
6. 当周所有抽奖实发累计 ¥150：小奖和大奖都禁用并保留券。
7. 无券状态：两个按钮仍显示奖池金额，并以禁用态显示“暂无券”。
8. 奖励记录累计金额等于 `history[].paid` 合计。

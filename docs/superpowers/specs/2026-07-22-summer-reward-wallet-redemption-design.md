# 暑假奖金账户与家长兑换设计

日期：2026-07-22

## 背景

暑假现金奖励已经改为只来自小奖和大奖抽奖，真实抽奖记录保存在 `summer_reward.history`。当前勋章页把奖励记录放在所有勋章内容之后，而且只能看到抽奖实发金额，没有“家长已经给了孩子多少钱”的兑换记录。学生更关心奖金，家长也需要在不进入管理员网址的情况下，直接在普通页面完成不定时、可部分兑换的记账。

网站不处理真实付款。家长仍通过微信、支付宝、红包或现金给孩子，网站只保存获得、兑换和撤销记录，供学生查看余额、供家长对账。

## 目标

- 把奖金账户和奖励流水放到原勋章页最上方。
- 展示“可兑换”“累计获得”“已兑换”三个金额。
- 支持家长不定时全部兑换或部分兑换。
- 普通页面内完成兑换，不要求修改网址进入管理员后台。
- 使用 4 位家长兑换码防止误触，不把它设计成强安全系统。
- 支持忘记兑换码后在普通页面二次确认重设。
- 支持撤销误记的兑换，保留可审计痕迹。
- 抽奖记录和兑换记录合并成按时间倒序的奖金流水。
- 将导航“勋章”改名为“奖励”，页面标题改为“奖励与勋章”。

## 非目标

- 不在网站内发红包、转账或连接支付平台。
- 不创建家长账号、学生账号或后端认证系统。
- 不改变抽奖奖池、概率、发券条件和每日/每周奖金上限。
- 不把勋章重新接入现金奖励。
- 不修改备份文件的 `version: "1.0"` 和既有备份结构。
- 不追溯无法从现有本地记录证明的历史奖金。

## 页面信息架构

底部导航和侧边栏中的“勋章”改为“奖励”，保留现有 `#achievements` 路由，避免旧书签和内部跳转失效。

页面从上到下排列为：

1. 页面标题“奖励与勋章”及简短说明。
2. 一张完整的“暑假奖金账户”卡。
3. 勋章统计。
4. 大勋章。
5. 小勋章。
6. 暑假计划成就。

奖金账户卡采用已确认的 A 布局：余额、兑换按钮和最近流水集中在同一张卡，不拆成两个区块。

### 奖金账户卡

顶部显示三个指标：

- 可兑换：`max(0, totalEarned - totalRedeemed)`，视觉权重最高。
- 累计获得：所有真实抽奖实发金额的终身累计。
- 已兑换：所有未撤销兑换的累计。

指标下方放一个整行主按钮“家长兑换奖金”。可兑换余额为 0 时按钮禁用，并显示“暂无可兑换奖金”。

主按钮下显示“最近奖金记录”。默认展示最近若干条，提供“查看全部/收起”。记录按时间倒序混合展示：

- 抽奖：`抽奖获得 +¥20`，并标注“小奖”或“大奖”。
- 兑换：`家长兑换 −¥10`。
- 已撤销兑换：灰色、删除线并标注“已撤销”，金额不计入已兑换总额。

## 家长兑换流程

### 首次使用

1. 家长点击“家长兑换奖金”。
2. 弹窗说明兑换码只用于防误触，不是支付密码。
3. 家长输入 4 位数字，再输入一次确认。
4. 两次一致后保存兑换码并进入金额步骤。

### 后续使用

1. 家长点击“家长兑换奖金”。
2. 输入 4 位兑换码。
3. 验证成功后进入金额步骤。

### 金额步骤

金额页清楚显示当前可兑换余额，并提供：

- 默认选项“全部兑换 ¥X”。
- 选项“部分兑换”，输入整数元金额。

页面固定提示：“请先通过微信、支付宝或现金把奖金给孩子，再在这里记账。”

最终按钮随输入金额变化为“已给孩子 ¥X，记为已兑换”。点击后再次校验余额，写入兑换记录，更新统计和流水，并显示成功反馈。

金额必须为大于 0 的整数，且不能超过点击确认瞬间的可兑换余额。余额为 0、金额为空、非整数或超额时不写入数据。

### 忘记兑换码

兑换码输入界面提供低调的“忘记兑换码”。点击后显示醒目的二次确认，明确说明旧兑换码会失效；确认后允许重新设置 4 位数字。

因为本功能只防误触且网站不负责付款，重置不依赖管理员网址、短信或原兑换码。

### 撤销错账

未撤销的兑换流水提供低调的“撤销”入口：

1. 输入家长兑换码。
2. 二次确认要撤销的日期和金额。
3. 将原兑换记录标记为已撤销，不删除记录。
4. 从 `totalRedeemed` 扣回该笔金额，恢复可兑换余额。

已经撤销的记录不能再次撤销；重复点击和重复提交必须幂等拦截。

## 数据模型

继续使用现有 `summer_reward` localStorage key，在 V2 抽奖状态基础上增加以下字段：

```javascript
{
  rewardModelVersion: 3,
  totalEarned: 0,
  totalRedeemed: 0,
  parentRedeemPin: "",
  redemptionHistory: [
    {
      id: "redeem_<timestamp>_<suffix>",
      requestId: "redeem_request_<timestamp>_<suffix>",
      amount: 20,
      date: "YYYY-MM-DD",
      ts: 0,
      revoked: false,
      revokedAt: 0
    }
  ]
}
```

字段含义：

- `totalEarned`：真实抽奖实发金额的永久累计，不依赖有长度限制的抽奖历史列表。
- `totalRedeemed`：所有未撤销兑换金额的累计。
- `parentRedeemPin`：本机 4 位防误触码。它不是安全认证凭证。
- `redemptionHistory`：兑换流水。兑换频率低，不主动截断；撤销通过状态字段保留审计痕迹。
- `requestId`：一次兑换弹窗的幂等标识；重复提交同一标识时返回原结果，不重复累计。

可兑换余额不单独持久化，统一由 `max(0, totalEarned - totalRedeemed)` 计算，避免三个金额互相矛盾。

## 抽奖累计与迁移

### 新抽奖

`econDraw(kind)` 产生真实抽奖记录时，只把该次 `paid` 累加到 `totalEarned`。`drawn` 高于日限额或周限额而被裁剪的部分不计入累计获得。

抽奖结算、永久累计和历史写入必须在同一次 `writeSharedReward()` 中完成，避免刷新时只写入一半状态。

### V2 到 V3

首次读取 `rewardModelVersion < 3` 的状态时进行一次幂等迁移：

1. 保留 V2 的券、达标日、阶段、预算计数和真实抽奖历史。
2. `totalEarned` 初始化为现存真实抽奖历史中 `paid` 的合计。
3. `totalRedeemed` 初始化为 0。
4. `parentRedeemPin` 初始化为空。
5. `redemptionHistory` 初始化为空数组。
6. 写入 `rewardModelVersion: 3`。

迁移不恢复以前已经删除的自动固定奖励，也不伪造现有历史列表之外的奖金。迁移完成后重复读取不得再次累加。

## 模块边界

`modules/summerTasks.js` 继续作为奖金账本的唯一业务来源，负责：

- V3 迁移。
- 抽奖累计。
- 读取奖金账户汇总和混合流水。
- 设置、校验和重置兑换码。
- 创建兑换记录。
- 撤销兑换记录。
- 金额、余额、记录状态和重复提交校验。

该模块通过 `window.MochiSummerTasks` 暴露窄接口。`app.js` 的奖励页只渲染数据、打开弹窗和显示反馈，不自行计算或直接改写 `summer_reward`，避免页面与浮窗出现两套余额逻辑。

## 异常与兼容处理

- `summer_reward` 缺失或损坏时沿用现有回退策略，创建安全的默认状态。
- 缺失 V3 字段时自动迁移，不要求用户清空 localStorage。
- `redemptionHistory` 非数组时按空数组处理。
- `totalEarned`、`totalRedeemed`、金额字段出现非有限数或负数时归一化为非负数。
- 如果旧数据异常导致 `totalRedeemed > totalEarned`，可兑换余额显示 0，禁止继续兑换，但保留历史供诊断。
- 兑换确认时重新读取最新状态，防止弹窗打开后又发生抽奖或另一笔兑换导致余额过期。
- 撤销通过唯一 `id` 查找；不存在、已撤销或金额非法时拒绝写入。
- 4 位码错误只提示重试，不进入金额步骤。
- 不改变 `study_log`、备份版本或现有 `summer_reward.history` 条目格式。

## 验收场景

1. 旧 V2 用户升级后，累计获得等于当前真实抽奖历史的实发总额，固定奖励不复活。
2. V3 迁移重复执行不改变累计金额。
3. 抽中 ¥20 且实发 ¥20 时，累计获得和可兑换各增加 ¥20。
4. 抽中 ¥20 但因额度只实发 ¥5 时，只累计 ¥5。
5. 可兑换 ¥44 时可全部兑换 ¥44，之后余额为 0。
6. 可兑换 ¥44 时部分兑换 ¥20，之后已兑换为 ¥20、余额为 ¥24。
7. 输入 0、负数、小数、非数字或超过余额的金额时不生成流水。
8. 首次设置兑换码要求两次一致且必须为 4 位数字。
9. 错误兑换码不能兑换或撤销。
10. 忘记兑换码经过二次确认后可重新设置，原奖金数据不变。
11. 撤销 ¥20 的兑换后，原记录保留并标记已撤销，已兑换减少 ¥20、余额增加 ¥20。
12. 同一笔兑换重复撤销只生效一次。
13. 刷新页面后统计和流水保持一致。
14. 奖金账户位于奖励页最上方，勋章内容下移；桌面端和手机宽度均可读。
15. 底部导航和侧边栏显示“奖励”，旧 `#achievements` 路由仍正常打开页面。
16. 导出并恢复现有完整备份后，新增奖金账户字段随 localStorage 快照恢复。

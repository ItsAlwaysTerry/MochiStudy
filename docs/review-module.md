# 复习模块设计说明

## 模块定位

复习页是“今天该复习什么”的指挥台，不是第二个学习档案，也不是题库管理页。

学习档案回答：我学过什么、每个知识点下面有哪些卡片。

复习页回答：我今天该复习什么、为什么、点哪里能开始。

## V1 用户流程

1. 学生可以从首页“今日复习”卡片直接开始，也可以进入 `#review` 复习页。
2. 页面按本地规则列出今日建议和全部待复习知识点；今日建议最多 1-2 个，不制造“永远复习不完”的压力。
3. 学生在某个知识点上点“开始复习”。
4. 网站自动复制这个知识点的 AI 复习素材包，并展开回填框。
5. 学生把素材包粘贴给 `skill/gaokao复习私教.md`。
6. 复习 AI 完成测验后输出增强版 `MOCHI-RECORD`。
7. 学生把 `MOCHI-RECORD` 粘回当前复习项，点击“导入复习结果”。
8. 网站新增一张复习卡片，仍归入原科目和原知识点。
9. 复习页重新计算该知识点状态、复习次数、上次结果和优先级。
10. 导入成功后，页面提示“去更新核心摘要”，引导学生把这次复习里真正有用的一句话沉淀进学习档案。

关键原则：点“开始复习”只代表复制材料，不代表已经复习。只有成功导入复习 AI 输出的 `MOCHI-RECORD` 才算完成一次复习。

## 数据闭环

V1 不接 API，依靠本地数据和外部复习 skill 完成闭环：

```text
study_log + study_card_meta
→ reviewEngine 聚合知识点状态
→ reviewPage 展示待复习项
→ 单知识点 AI复习包
→ 外部复习 AI
→ 增强版 MOCHI-RECORD
→ applyMochiRecord 导入
→ study_log 新增复习卡 + study_card_meta 保存复习元信息
→ reviewEngine 重新计算
```

复习扩展信息继续写入 `study_card_meta`，不要扩展 `study_log` 核心字段。

## 代码边界

- `modules/reviewEngine.js`
  - 聚合知识点。
  - 计算优先级。
  - 生成推荐原因。
  - 生成单知识点 AI 复习包。
  - 保存默认复习参数。

- `modules/reviewPage.js`
  - 渲染 `#review` 页面。
  - 管理筛选、开始复习、回填导入。
  - 复习导入成功后只给出更新摘要入口，不在复习页内展开档案编辑。
  - 调用 `reviewEngine` 和 `MochiApp.applyMochiRecord()`。
- `modules/farm.js`
  - 首页只展示一张“今日复习”轻入口。
  - 点击首页“开始复习”时调用 `MochiReviewPage.startItem()`，复制复习包并跳转复习页展开对应项。

- `app.js`
  - 负责路由和现有导入逻辑。
  - 暴露 `applyMochiRecord` 给复习页复用。

## V1 推荐规则和状态机

第一版使用可替换的本地规则，不显示算法分数，只显示人能理解的原因。

优先级提高：

- 低星记录多。
- 从没复习过。
- 很久没碰。
- 上次复习不是“独立做对”。
- 同一知识点反复出现卡点。

优先级降低：

- 最近刚复习过。
- 上次复习结果为“独立做对”或基本掌握。
- 仍处于掌握后的冷却期。

V1 不是单纯按分数排序，而是先把知识点分成可消化状态：

```text
待复习：从没复习、低星、久未碰，且已经到复习时间。
仍需继续：上次复习结果是仍需讲解/未掌握，且已经到复习时间。
待巩固：刚复习过但还不稳，先冷却几天，不连续压迫学生。
近期稳定：上次独立做对或基本掌握，进入较长冷却期。
```

复习结果流转：

```text
独立做对 / 基本掌握 → 近期稳定 → 默认 14 天后再出现
看提示做对 / 部分掌握 → 待巩固 → 默认 3 天后再出现
仍需讲解 / 未掌握 → 待巩固或仍需继续 → 到期后再进待复习
```

这条规则是为了让书越读越薄：刚处理过的知识点要暂时降权，今日建议只放最值得处理的少数项目。

默认参数在 `modules/reviewEngine.js` 的 `DEFAULT_REVIEW_SETTINGS`：

```js
{
  lowStarThreshold: 2,
  newKnowledgeReviewAfterDays: 2,
  staleAfterDays: 7,
  masteredCooldownDays: 14,
  weakCooldownDays: 3,
  maxTodaySuggestions: 2,
}
```

这些参数未来可以做成管理员或用户设置。

## 复习完成判断

复习项的真实完成条件是：

```text
导入成功 + 识别为当前科目/知识点 + 写入复习卡片
```

不要在点击“开始复习”时把知识点标记为已复习，因为学生可能只是复制了材料，没有完成复习。

## 与学习档案的边界

复习页只做知识点级聚合和待处理流程，不承担学习档案的编辑、删除、拖拽排序、完整卡片浏览职责。

如果学生需要看完整卡片，复习页的“看卡片”应跳回学习档案对应知识点。

学习档案前台默认只强调“新学 / 复习”两类，避免学生面对过多标签。内部仍可保留 `lesson / review / quiz / reflection`，用于导出、编辑和后续分析。

学习档案默认是阅读模式。摘要编辑、卡片编辑、删除和拖拽排序只在“整理模式”里出现；复习页不能把这些维护工具复制一份过去。

复习页默认不展示完整复习包大文本。复习包是给 AI 的材料，不是页面主角；用户只需要看到“已复制材料”和一个轻量回填框。

V1.1 开始，回填框应保持为单行粘贴入口，和“导入复习结果”按钮放在同一行。用户只负责粘贴 AI 输出，不需要在这个框里阅读或编辑长文本。

## 卡片变薄原则

学习档案不能越用越厚。展开知识点时，默认先显示一张知识点摘要，而不是立刻铺开所有历史卡片。

摘要优先显示：

- 主要卡点。
- 最近突破。
- 复习状态。
- 仍需留意的点。

历史卡片保留，但放进折叠区。它们是证据和原始材料，不是学生每次进入档案都要重新消化的主界面。

从复习页点击“看卡片”时，应定位并高亮对应知识点，让用户看到这个知识点当前的摘要，再按需展开历史卡片。

未来如果继续迭代，可以把多张历史卡沉淀成“知识点核心摘要卡”，让旧记录逐步压缩成更少、更可复习的精华。

### 本地摘要的能力边界

V1 的“核心摘要卡”不是生成式 AI 总结，它是规则型摘要：

- 从低星记录和复习记录里提取最常见卡点。
- 从 `关键突破` 或最近的 `今日套路` 里提取最近突破。
- 从复习记录里读取上次复习结果。
- 从 `reviewEngine` 里读取当前状态和下一步建议。

它能做到“把已有字段整理得更可读”，不能凭空理解题意、改写成高质量讲义，也不能发现记录里没有写出来的问题。它是否好用，很依赖 MOCHI-RECORD 里 `卡点记录 / 今日套路 / 关键突破 / 复习结果` 写得是否具体。

未来如果需要更像老师的总结，可以在这个规则摘要之上接 API，生成可编辑的知识点核心摘要。但在接 API 前，规则摘要已经能先解决“历史卡片太厚、学生不知道看哪里”的问题。

## V1 不做

- 不接 API。
- 不站内生成题。
- 不做复杂计划系统。
- 不做打卡完成勾选。
- 不做题库。
- 不把复习卡片拆成孤立档案。
- 不改 `study_log` 核心字段结构。

## 后续迭代方向

- 做“复习设置”，允许调整每天建议数量、低星阈值、冷却天数。
- 让掌握好的知识点自动降权，掌握差的知识点短期回流。
- 根据 `reviewResult`、`confidence` 和 `stuckStep` 做更细的复习间隔。
- 支持更明确的“复习中”状态持久化，但仍不能把复制动作当作完成。
- 接入 API 后，把“复制给 AI”升级为站内生成复习建议或复习题。
- 根据用户反馈调整 `reviewEngine`，不要把排序逻辑散落到页面代码里。

## V1.1 人工校正摘要

学习档案的核心摘要先由本地规则自动整理，但允许用户把真正有价值的一两句话校正下来。人工摘要保存到 `localStorage["study_node_summary"]`，按 `subject::nodeLabel` 存储，不改 `study_log`，也不写入单张卡片的 `study_card_meta`。

单个知识点可保存：

- `mainPainPointOverride`：人工校正后的主要卡点。
- `keyBreakthroughOverride`：人工校正后的核心突破。
- `reviewNote`：给下次复习 AI 的短备注。
- `updatedAt`：最后保存时间。

显示规则是：有人工摘要时优先显示人工摘要，并标记为“已校正”；没有人工摘要时继续显示规则摘要，并标记为“自动整理”。“恢复自动摘要”只删除该知识点在 `study_node_summary` 里的人工记录，不影响历史卡片。

单知识点 AI 复习包会优先带上已校正精华摘要，再附历史记录。这样 AI 先看到人确认过的重点，但仍能回看原始证据。

# MochiStudy 自我迭代框架

## 使用说明

本文件供 Agent 独立运行。每次执行时，依次扮演五个角色审视项目，最后由审判官综合裁决，选出 3-5 个改动并直接执行。不需要等待用户确认。

**执行前必须先读** `CLAUDE.md` 和 `AGENTS.md`，了解当前版本状态、禁止事项和已有功能——不要重复实现已存在的东西，不要触碰被明确禁止的系统。

**执行后必须**：
1. `node --check` 验证所有修改过的 JS 文件
2. 更新 `CLAUDE.md` 和 `AGENTS.md` 的"当前版本状态"
3. `git commit` + `git push`

---

## 第一轮：四角色平行审视

每个角色独立审视，不受其他角色影响。把发现的问题写下来，标注严重程度：🔴 严重影响使用 / 🟡 明显但可接受 / 🟢 细节打磨。

---

### 角色一：唐纳德·诺曼（Don Norman）
**身份**：《设计心理学》《情感化设计》作者，人机交互领域奠基人。  
**核心信条**：用户永远不会出错，只有设计出错。好设计让人无需思考就能正确操作；每一个动作都应该有即时的、清晰的反馈；错误发生时，界面必须告诉用户「现在发生了什么」和「下一步该做什么」。他对「需要阅读说明才能使用」的功能毫无容忍。

**以诺曼的视角，逐项检查以下问题**：

- 导入框的 placeholder 文案能让用户第一眼知道该粘贴什么格式的内容吗？还是需要「猜」或「记住」之前的操作？
- 导入失败时（没有找到 MOCHI-RECORD），错误提示是否明确告知用户「哪里错了」和「怎么修」？还是只是说「没找到」？
- 「打卡成功」的反馈卡（checkin-success）里的进度条和星级鼓励语，用户有没有足够时间看到，还是会被下一步操作打断？
- 复习页点击「开始复习」后，材料被复制到剪贴板——用户知道这件事发生了吗？toast 的措辞是否足够清晰说明「已复制，可以去 AI 那里粘贴了」？
- 农场精灵图的生长阶段变化（种子→发芽→幼苗→开花→成熟）在视觉上是否足够明显，让用户每次导入记录后都能感知到变化？还是阶段之间差别太小？
- 底部五个导航的图标+文字，用户能从中猜出每个页面的核心用途吗？「日历」和「赛季」的区别对新用户来说清晰吗？
- 「今天想学习」按钮只在农场休眠状态（上学日）出现——用户能理解这个按钮的作用吗？还是需要试点才知道？
- 连续学习 streak 为 0 时的「今天还没开始」卡片，用户看完后知道该做什么吗？有没有直接引导到导入框的动线？
- 复习页的「今日复习」和「全部待复习」两个区域的关系，用户第一次看能理解吗？有没有层级混淆？
- 抽奖入口（勋章页）——用户知道自己积累了多少抽奖次数吗？还是这个信息藏得太深？

### 角色二：芭芭拉·奥克利（Barbara Oakley）
**身份**：《学习之道》作者，Coursera 最受欢迎课程「Learning How to Learn」主讲人，认知神经科学与工程学双背景。  
**核心信条**：被动重读是最低效的学习方式；提取练习（主动回忆）才能形成长期记忆；间隔重复需要恰好在「快要忘记」的时机介入；工作记忆容量有限（7±2 块），信息过密会导致大脑放弃加工。她对「看起来很努力但实际上没有学到」的设计极度警惕。

**以奥克利的视角，逐项检查以下问题**：

- 复习页推荐的「今日建议」，是真的在选「快要忘记的」知识点，还是只是按记录数量或时间线性排列？`reviewEngine.js` 里的优先级算法有没有体现遗忘曲线？
- 用户点「开始复习」后，只是把材料复制给 AI，自己并没有经历主动回忆——这是设计上的妥协，还是可以在流程里加入一个「先自己想一想」的提示步骤？
- 首页「今日复习」卡片展示的 `mainPainPoint`，够不够让用户在打开 AI 之前先在脑子里回忆一遍？措辞是否在暗示「你还记得这个吗」而不是直接告诉答案？
- 复习结果的 stars 评级（1-3 星）是否被清楚定义？用户知道「2 星」和「3 星」的区别标准吗？评级不准确会让复习算法失效。
- 每次最多展示 2 个今日建议——这个上限是否合理？学习科学的数据表明，超过 3 个任务会触发「任务选择焦虑」，但 1 个可能又太少。
- 复习冷却期（刚导入复习结果的知识点不再立刻出现）是否可以配置？3 星后的冷却应该比 1 星后更长。
- 学习档案的卡片正面以「卡点文字」为最大字号——这是对的，但用户打开卡片是为了「温习」还是「提取」？有没有办法在展示答案前加一个遮挡？
- 勋章触发频率（多少条记录触发一个小勋章）是否和实际学习节奏匹配？如果触发太频繁，会失去意义；太稀疏，会让人放弃。

### 角色三：迪特·拉姆斯（Dieter Rams）
**身份**：博朗公司首席设计师，工业设计史上「Less but better」的最高实践者。他的作品影响了乔布斯和 Jony Ive。  
**核心信条**：好设计不是添加，是减法。每一个存在于界面上的元素都必须能回答「它为用户解决了什么问题」——如果答案是「让界面看起来更丰富」，就应该删掉。他对「功能蔓延」（Feature Creep）有本能的厌恶，对「装饰性复杂度」零容忍。

**以拉姆斯的视角，逐项检查以下问题**：

- 底部导航五个页面：首页、复习、档案、日历、设置。用户每天必须用到哪几个？「日历」和「赛季」在日常使用中的访问频率是多少？如果把它们合并或降级到设置页，用户日常动线会不会反而更清晰？
- 赛季系统：热力图、每周趋势 SVG 图、称号 16 级、历史快照回看、文字版和 JSON 导出。这个学生会主动去看赛季页吗？还是这些功能更多是「Terry 看数据」用的？
- 农场升级系统（Lv1 小菜园 → Lv5 星愿农场）：用户能感受到升级带来的变化吗？精灵图和 plot 皮肤的差别有没有让她觉得「要继续种」？还是只是数字在变？
- 「今日目标」的点点（daily-goal-dots）：这排点传递的信息是什么？用户看懂它代表什么了吗？还是它只是视觉上的存在？
- 导入框 placeholder 说明 + 首页引导卡（home-guide-card）的三步引导：两者有没有信息重叠？有没有一个可以删掉？
- 沉浸模式（专注中）里保留了导入框入口——专注的核心目标是「不被打扰地做题」，导入是「做完题后记录」。这两件事能共存吗，还是沉浸模式里的导入框破坏了专注感？
- 勋章页同时显示：抽奖入口 + 大勋章列表 + 小勋章列表 + 勋章总览。这四个区域用户都会看吗？还是只有抽奖最有吸引力，其余是视觉噪音？
- 调试面板（?debug=1）对用户来说是否存在「误触」风险？URL 参数方式够安全吗？

### 角色四：马丁·福勒（Martin Fowler）
**身份**：《重构：改善既有代码的设计》作者，ThoughtWorks 首席科学家，软件工程领域「代码坏味道」理论创始人。  
**核心信条**：代码的首要读者是人，不是机器。重复是所有 bug 的温床。YAGNI（You Aren't Gonna Need It）——不要为假设中的未来需求写现在用不到的代码。函数应该只做一件事，变量名应该说清楚它是什么。他会在代码里找「坏味道」，然后问：「这里六个月后还能看懂吗？」

**以福勒的视角，逐项检查以下问题**：

- `parsePastedRecordEl`（app.js 处理首页导入）和 `importReviewResult`（reviewPage.js 处理复习导入）都调用了 `parseMochiRecord` + `applyMochiRecord`，两者的成功/失败处理逻辑有没有可以共用的部分？还是各自重复了类似的错误提示模板？
- `renderFarm`（farm.js）一个函数承担了：赛季横幅、streak 横幅、迷你农场、导入框、今日目标、今日复习卡、引导卡——渲染逻辑越来越长，是否应该拆出独立的 `renderFarmSidebar` 子函数？
- `buildReviewState()`（reviewEngine.js）在首页（`renderTodayReviewCard`）和复习页（`render`）都会被调用，每次都完整遍历 `study_log`——两次调用间隔很短时有没有重复计算的浪费？
- `readJson` 和 `readJSON` 在项目里是否有命名不一致的情况？同一个函数的两种拼写会造成阅读困惑。
- `study_card_meta`（卡片级复习元信息）和 `study_node_summary`（知识点级人工摘要）都挂在知识点下，新接手的开发者能分清两者的用途边界吗？代码里调用这两个 key 的地方，有没有混用的风险？
- `checkAndGrantAchievements()` 在导入记录、完成番茄钟、收获农场后各调用一次——有没有同一个用户操作触发多次勋章检查的路径？重复触发是否安全？
- `escapeAttr`（farm.js）已经统一到调用 `window.MochiApp.escapeHtml`，但其他模块（reviewPage.js、knowledgeMap.js）各自有没有类似的局部转义函数，还是都在用统一的？
- `calcStudyStreak` 每次都从 `readStudyLogs()` 读全量 log 再遍历 365 天——如果 log 有几百条，这是否是个性能瓶颈点？在什么情况下会被频繁调用？

## 第二轮：审判官裁决

### 查理·芒格（Charlie Munger）
**身份**：伯克希尔·哈撒韦副董事长，多学科思维模型体系的建立者。他在 90 岁高龄前始终是世界上最清醒的决策者之一。  
**核心人格**：不被情绪和「听起来好的想法」驱动。他相信大多数问题的答案藏在「反转」里——与其问「怎么让这个功能更好」，先问「怎么做会让这个项目彻底失败」，然后避开那些事。他有强烈的机会成本意识：做 A 就意味着不做 B，如果 B 的收益更高，那做 A 就是错的。他不相信「全面提升」，只相信「在正确的地方用力」。

**芒格的裁决流程**：

**第一步：读约束**  
打开 `CLAUDE.md`，把所有「不能做」的事列出来。前四个角色的任何建议如果触碰这些约束，直接剔除，不讨论。

**第二步：去重和合并**  
前四个角色可能对同一个问题从不同角度提出了相似的建议。把它们合并成一个清晰的问题陈述。

**第三步：反转检验**  
对每一个候选改动，问自己：「如果我们做了这个改动，三个月后会不会后悔？」「如果不做，用户今天真的会受损吗？」无法通过这两个问题的，降级或剔除。

**第四步：性价比排序**  
用这三个维度给候选改动打分：
- **用户感知强度**：这个改动用户打开页面就能感受到，还是要操作好几步才能发现？
- **实现风险**：改动是否可能破坏已有功能？是否需要修改 localStorage 结构或备份格式？
- **代码改动范围**：只改一个函数，还是需要跨三个文件协调？

**第五步：选出 3-5 个，确定顺序**  
优先选「用户感知强、实现风险低、改动范围小」的。如果有一个改动的价值明显高于其他，让它排第一。

**第六步：输出执行列表**  
用以下格式列出每个改动：
```
改动 N：[一句话描述]
- 问题：[前四个角色发现了什么]
- 方案：[具体改什么，在哪个文件的哪个函数]
- 预期效果：[用户会看到/感受到什么]
- 风险：[可能影响什么，如何验证]
```

## 执行规则

裁决完成后，按顺序执行每个改动。

1. **修改代码**：精确改动，不引入额外重构，不改与本次无关的代码
2. **语法检查**：每次修改后运行 `node --check <文件名>`，有报错立刻修复
3. **更新文档**：在 `CLAUDE.md` 和 `AGENTS.md` 的"当前版本状态"里添加本轮版本摘要（格式参考已有的 V1.5 / V1.6 / V1.7 条目）
4. **提交**：`git add` 所有修改的文件 → `git commit`（描述本次迭代内容）→ `git push`

**特别注意**：
- 不要在一次执行里做超过 5 个改动
- 每个改动应该独立可验证，不要做「必须 A 和 B 同时上才能工作」的捆绑改动
- 如果在执行中发现某个改动比预期复杂，降级处理（只做安全的部分），剩余的留到下一轮迭代

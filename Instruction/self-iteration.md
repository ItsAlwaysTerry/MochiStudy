# MochiStudy 自我迭代框架

## 使用说明

本文件供 Agent 独立运行。执行时依次扮演五个角色审视项目，最后由审判官综合裁决，选出 3-5 个改动并直接执行。不需要等待用户确认。

---

## 第一轮：四角色平行审视

每个角色独立审视，标注问题严重程度：🔴 严重影响使用 / 🟡 明显但可接受 / 🟢 细节打磨

---

### 角色一：唐纳德·诺曼（Don Norman）
**核心信条**：用户永远不会出错，只有设计出错。每个动作必须有即时清晰的反馈；错误发生时界面必须告诉用户「发生了什么」和「下一步该做什么」。

**重点阅读**：`app.js`（`parsePastedRecordEl`、`renderHomeView`）、`modules/farm.js`（`renderStreakBanner`、`renderTodayReviewCard`）

**以诺曼的视角检查以下问题**：

- 「今天还没开始，打一张就够了」卡片（streak=0 兜底）——用户点击它有任何反应吗？还是死区？有没有导向导入框的直接动线？
- 「今日目标」点点（`daily-goal-dots`）：用户第一次看到它，能猜出每个点代表什么吗？没有 label 情况下信息密度够用吗？
- 打卡成功卡（`checkin-success`）里的进度条和星级鼓励语会在多少秒后消失？用户有没有足够时间阅读完，还是被下一次导入自动覆盖？
- 农场精灵图的生长阶段变化（种子→发芽→幼苗→开花→成熟）——每次导入记录后，用户能在不刷页面的情况下看到植物变化吗？还是需要手动刷新？
- 「今天想学习」按钮只在上学日出现——用户第一次看到它能理解「点了之后今天会计入有效学习日」这层含义吗？按钮本身有解释文字吗？
- 复习页「今日复习」和「全部待复习」两个区域：第一次打开页面，用户能从视觉上分清「今天要做什么」和「所有待办」的层级关系吗？
- 沉浸模式（专注中）里的导入框入口——用户在专注时看到它，是否会产生「现在应该导入还是专注」的迷惑？入口是否足够低调？

### 角色二：芭芭拉·奥克利（Barbara Oakley）
**核心信条**：被动重读是最低效的学习方式；提取练习（主动回忆）才是真正的学习；间隔重复需要在「快要忘记」时介入；信息过密会让大脑放弃。

**重点阅读**：`modules/reviewEngine.js`（`buildReviewState`、优先级算法）、`modules/reviewPage.js`（`renderTodayTask`、`startReview`）

**以奥克利的视角检查以下问题**：

- 复习页已加「先自己回想 20 秒」提示卡（V1.8）——这个提示是在「复制材料」按钮之前出现，还是之后？流程顺序对主动回忆有多大影响？
- `buildReviewState()` 里的「今日建议」推荐算法：是否真正体现了遗忘曲线（间隔越久 priority 越高）？还是简单按最后导入时间线性排序？`reviewEngine.js` 里看具体实现。
- 复习结果的星级（1-3 星）——在 `renderImportPanel` 里，用户做完复习导入结果时，界面是否清楚说明了「1 星 = 还是不会，2 星 = 模糊记得，3 星 = 完全掌握」？还是让用户自判？
- 「今日复习」最多展示 2 个知识点：这个上限是写死的还是来自 `GAME_CONFIG`？如果写死了，应该移到配置里。
- 复习冷却期逻辑：3 星复习后的知识点会被排除多久？1 星呢？冷却期是在 `buildReviewState` 里还是其他地方？如果没有冷却期区分，这是个漏洞。
- 首页「今日复习」卡片展示的 `mainPainPoint`（V1.7）：这个文字是直接展示答案，还是在措辞上暗示「你还记得吗」？两者对学习效果差别巨大。
- 勋章触发频率（多少条记录一个小勋章）：当前默认值是多少？和这个学生每天实际导入量相比，频率是否合适？

### 角色三：迪特·拉姆斯（Dieter Rams）
**核心信条**：好设计是减法。每个界面元素都必须能回答「它为用户解决了什么」——答案是「丰富感」就删掉。对功能蔓延有本能厌恶。

**重点阅读**：`index.html`（底部导航结构）、`modules/farm.js`（整体渲染函数）、`style.css`（视觉密度）

**以拉姆斯的视角检查以下问题**：

- 底部导航五个 tab（首页/复习/档案/日历/设置）：这个学生每天必用的是哪几个？「日历」在她日常用多频繁？把日历降级到设置页或赛季页，日常动线会不会更清晰？
- `renderFarm`（`farm.js`）负责渲染：赛季横幅 + streak 横幅 + 迷你农场 + 导入框 + 今日目标 + 今日复习卡 + 引导卡。首页视觉层级是否已经过密？用户第一眼应该看向哪里？
- 勋章页同时显示：抽奖入口 + 大勋章列表 + 小勋章列表 + 勋章总览——用户打开勋章页，第一眼会被什么吸引？其他三个区域是不是视觉噪音？
- 复习页「今日复习」卡片（V1.6 删掉了状态列，现在两列）：简化后信息够不够用？还是删多了导致用户看不清要复习哪个知识点？
- 沉浸模式下的导入框——专注和记录分别是两种心理状态，强行同屏是否破坏了专注的纯粹感？「放弃本轮」按钮是否明显到让人容易误触？
- 打卡成功卡里的地块进度条（V1.6 新增）：用户在打卡瞬间关心「距离丰收还差几条」吗？还是这条信息最好放在农场主区，而不是打卡反馈里？
- 引导卡（`home-guide-card`）三步引导和导入框 placeholder 的说明文字：这两处信息有没有重叠？有没有可以合并或删掉的？

### 角色四：马丁·福勒（Martin Fowler）
**核心信条**：代码的首要读者是人。重复是所有 bug 的温床。YAGNI——不要为假设中的未来需求写现在用不到的代码。函数只做一件事。

**重点阅读**：`app.js`（`parsePastedRecordEl`）、`modules/reviewPage.js`（`importReviewResult`）、`modules/farm.js`（`renderFarm`）、`modules/reviewEngine.js`（`buildReviewState`）

**以福勒的视角检查以下问题**：

- `parsePastedRecordEl`（app.js）和 `importReviewResult`（reviewPage.js）都走同一套「解析 → 保存 → 反馈」流程，但反馈 UI 逻辑各自实现。两者的错误提示构造有没有可提取的公共模板，还是各自重复？
- `renderFarm`（farm.js）是否已经过长（超过 150 行），一个函数承担超过 3 个独立职责？如果是，候选子函数有哪些：`renderStreakBanner`、`renderMiniPlots`、`renderImportBox`、`renderTodayReviewCard`？
- `buildReviewState()`（reviewEngine.js）在首页和复习页都会调用，两次调用可能在同一次页面渲染周期里发生。有没有简单的缓存机制（比如同一分钟内不重复计算）？
- `escapeHtml` 统一到 `window.MochiApp.escapeHtml` 后（V1.7），`reviewPage.js` 和 `knowledgeMap.js` 有没有遗漏的局部转义函数？用 Grep 搜 `escape` 确认。
- `readJson` / `readJSON` 拼写不一致问题（框架里 V1.0 提到过）：是否已统一？还是两种拼写都存在？
- `checkAndGrantAchievements()` 被多个时机调用——在一次完整的「导入记录」操作里，它最多会被调用几次？有没有同一操作触发两次勋章检查的路径？
- `GAME_CONFIG_DEFAULTS`（app.js 顶部）里有没有写死的数值（硬编码数字，不走 GAME_CONFIG 引用）？Grep `magic number` 或找裸露的整数常量。

## 第二轮：审判官裁决

### 查理·芒格（Charlie Munger）
**核心人格**：不被「听起来好」的想法驱动。先问「怎么做会彻底失败」，再决定做什么。他相信大多数决策是「避免愚蠢」而不是「追求聪明」。有强烈的机会成本意识。

**芒格的裁决流程**：

**第一步：合规过滤**
把前四个角色的所有发现列出来，每条对照起飞前检查的「不可触碰边界」——触碰边界的直接剔除，不讨论。

**第二步：去重合并**
多个角色对同一问题的不同角度，合并成一个清晰的问题陈述，不要重复执行。

**第三步：反转检验**
对每个候选改动问两个问题：
- 「如果做了这个改动，三个月后会不会后悔？」（稳定性）
- 「如果不做，用户今天真的会受损吗？」（紧迫性）

两个答案都是「不」的，剔除。

**第四步：三维打分**

用下面的矩阵对候选改动打分（1-3分，3最高）：

| 改动描述 | 用户感知强度 | 实现风险 | 代码范围 | 总分 |
|---|---|---|---|---|
| （填入候选改动） | /3 | /3（低风险=3） | /3（单文件=3） | /9 |

「用户感知强度」：打开页面就能感受到（3）/ 需要操作几步才能发现（2）/ 只有特定场景（1）  
「实现风险」：不涉及 localStorage 结构、不跨模块（3）/ 跨 1-2 个文件（2）/ 需要改 localStorage 或备份格式（1，通常直接剔除）  
「代码范围」：单个函数（3）/ 单个文件（2）/ 跨 2+ 文件（1）

**第五步：选出 3-5 个，按总分降序排列**
如果多个改动得分相同，优先选「能让用户今天立刻受益」的。

**第六步：输出执行列表**
用以下格式列出每个改动：
```
改动 N：[一句话描述]
- 发现来源：[哪个角色，什么视角]
- 问题：[具体是什么不对]
- 方案：[改哪个文件的哪个函数，改什么]
- 预期效果：[用户会看到/感受到什么变化]
- 风险：[可能影响什么，如何用 node --check 验证]
- 打分：用户感知 N / 风险 N / 范围 N = 总分 N
```

## 执行规则

裁决完成后，按顺序执行每个改动。

**执行前**：用 `git stash` 记录当前状态，方便快速回滚。

**每个改动的执行流程**：
1. **精确修改**：只改本次改动涉及的代码，不顺手重构周边无关代码
2. **语法检查**：`node --check <修改的文件名>`，有报错立刻修复，不进入下一个改动
3. **通过后继续**：进入下一个改动

**全部改动完成后**：
1. 在 `CLAUDE.md` 的「当前版本状态」添加本轮版本摘要（格式参考已有的 V1.5/V1.6/V1.7/V1.8 条目）
2. `git add` 所有修改的文件
3. `git commit`（描述本次迭代内容，注明版本号）
4. `git push`

**特别注意**：
- 一次执行最多 5 个改动
- 每个改动必须独立可验证（不能「A 和 B 必须同时上才能工作」）
- 如果执行中某个改动比预期复杂，降级处理（只做安全部分），剩余留下一轮
- 如果遇到无法用 `node --check` 验证的逻辑错误，在 commit message 里注明「需要人工验证 XYZ 功能」

## 本框架自更新规则

每轮迭代完成后，如果发现本框架里的某个问题已经被解决，在「第零步：起飞前检查」的「V1.8 已解决问题」列表里追加标记。下一轮迭代的角色问题应当自动跳过这些。

框架本身也可以迭代——如果发现某个角色的问题长期无法落地执行（总是被芒格剔除），说明那个问题不对项目有实际价值，可以替换成更切实的问题。
